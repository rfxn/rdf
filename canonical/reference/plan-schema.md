# Plan Schema

Authoritative validation rules for PLAN.md. Single source of truth cited by
`/r-plan` (pre-reviewer pass), the `reviewer` agent (challenge mode), and
`/r-build` (Section 1, pre-dispatch).

A plan that violates any rule below is rejected. Validators stop at the
first violation and report which rule failed.

---

## Plan-Version Awareness

Before applying any rule below, check the plan preamble for:

```
**Plan Version:** 3.0.6
```

(or higher).

- **Marker present** — apply all rules.
- **Marker absent** (legacy pre-3.0.6 plan) — apply Rules 1-3 only; skip
  Rules 4-7 (Regression-case checks) with INFO log: *"Legacy plan detected
  (no Plan Version marker); Regression-case not enforced."*

---

## Rule 1: Phase Heading Shape

Every phase begins with a level-3 heading:

```
### Phase N: <description>
```

Where `N` is an integer ≥ 1 and `<description>` is non-empty.

**Failure:** *"Phase heading malformed at line <L>: expected `### Phase N: <description>`."*

---

## Rule 2: Mandatory Phase Metadata Fields

Every phase must contain all five fields, each exactly once:

| Field              | Allowed values                                                       |
|--------------------|----------------------------------------------------------------------|
| `**Mode**`         | `serial-context` \| `serial-agent` \| `parallel-agent`               |
| `**Accept**`       | Non-empty acceptance criteria (concrete, testable)                   |
| `**Test**`         | Test file + test names, or verification commands with expected output |
| `**Edge cases**`   | Spec edge cases covered by this phase, or literal `none`             |
| `**Regression-case**` | See Rules 3-7                                                     |

**Failure:** *"Phase N missing required field: <field>."* — list every
missing field, do not stop at the first.

---

## Rule 3: Phase Crash-Safety Marker

Every phase ends with a `---` horizontal rule before the next phase heading
(or EOF). A phase without a trailing `---` is **truncated** and will be
regenerated by `/r-plan --resume`.

**Failure:** *"Phase N is truncated (missing trailing `---`)."*

---

## Rule 4: Regression-case Shape

The `**Regression-case**` field value must match one of two shapes:

### 4a. Named-test reference

```
<path>::@test "<name>"          # BATS
<path>::<function_name>         # pytest, go test
```

`<path>` is a relative path to a test file that exists at plan-write time
**or** is created as part of the same phase. `# TODO: add test` is
rejected.

### 4b. Closed-category N/A

```
N/A — <category> — <reason>
```

`<category>` ∈ `{docs, performance, logging, refactor, security}` (see
Rule 5). `<reason>` is a non-empty justification.

**Both shapes:** the em-dash separator `—` (U+2014) is the canonical form.
ASCII double-hyphen `--` is also accepted. Single hyphen `-` is rejected
(too easy to confuse with ranges).

**Failure (named-test):** *"Regression-case '<value>' does not match a
known test reference pattern. Expected `<path>::@test \"<name>\"` (BATS)
or `<path>::<function>` (pytest/go)."*

**Failure (N/A):** *"Regression-case '<value>' does not match `N/A —
<category> — <reason>`."*

### 4c. Forbidden shapes

The following are common author mistakes — reject explicitly:

- **Whole-file invariant** (e.g., *"tests/foo.bats — existing G1-G15 tests
  must remain green"*). A regression case must pin to one named test.
  Use Rule 4a with a single chosen test, or Rule 4b with category
  `refactor` and a reason that references the file.
- **Deferred reference** (e.g., *"see CHANGELOG entry for issue refs"*,
  *"link TBD"*). The validator does not follow forward references —
  inline the CVE/bug/issue identifier in the reason field.
- **Free-form prose** without `::` separator (named-test) or without two
  `—` separators (N/A).

**Failure:** *"Regression-case '<value>' uses a forbidden shape: <which>.
See plan-schema.md Rule 4c."*

---

## Rule 5: Category Allowed Set

When Rule 4b applies, `<category>` must be one of:

| Category      | Use when                                                                       |
|---------------|---------------------------------------------------------------------------------|
| `docs`        | Documentation-only change (markdown, comments, README, CHANGELOG)              |
| `performance` | Optimization or perf tuning where regression is performance, not correctness    |
| `logging`     | Log-format or instrumentation change with no functional behavior delta          |
| `refactor`    | No production behavior change — includes pure test additions, code reorganization, internal renames |
| `security`    | Security fix where no single named test pins the regression (requires Rule 7) |

**Failure:** *"Regression-case category '<value>' not in allowed set
{docs, performance, logging, refactor, security}."*

---

## Rule 6: Category Disambiguation (common author mistakes)

The validator cannot mind-read intent, but the following heuristics flag
the most common misuses and are MUST-FIX when detected:

- **Pure test-addition phase ⇒ `refactor`, never `security`.** A phase
  that adds or modifies test files only (no production source change)
  uses category `refactor`. Reserve `security` for *security fixes* whose
  regression cannot be pinned to one named test.
- **Halo-category mistake.** A phase that *touches security-relevant
  paths* (auth, crypto, allowlists) but doesn't *fix a security issue*
  is not a `security` regression case. Pick the category that matches
  the change, not the surrounding code.
- **`refactor` requires no behavior change.** If the phase changes
  observable behavior, pick `performance`, `logging`, or use Rule 4a
  (named test).

**Failure:** *"Phase N: category `<value>` likely incorrect — <reason>.
See plan-schema.md Rule 6."* Reviewer flags as MUST-FIX; engineer resolves
by either correcting the category or adding a justification that overrides
the heuristic in the reason field (e.g., *"category security retained
because this test exercises CVE-2025-1234 mitigation despite being a
test-only phase"*).

---

## Rule 7: Security Category Requires Identifier

When `<category>` is `security`, the `<reason>` field must contain at
least one of:

- **CVE identifier:** `CVE-YYYY-NNNNN` (4-digit year, 4-7 digit ID)
- **Internal issue reference:** `#<digits>` (e.g., `#142`)
- **Upstream bug reference:** URL or `<project>#<digits>` form

**Failure:** *"Regression-case security N/A requires CVE/bug/issue
reference in reason. None found in '<reason>'."*

## Validator Behavior Summary

| Caller                | When it runs                          | On failure                                  |
|-----------------------|---------------------------------------|---------------------------------------------|
| `/r-plan` Step 3.0    | After PLAN.md write, before reviewer  | Report violations to user; halt before review |
| `reviewer` (challenge)| During plan review                    | Emit MUST-FIX finding citing the rule      |
| `/r-build` Section 1  | After PLAN.md read, before dispatch   | Print error, stop without dispatching       |

All three callers apply the same rules in the same order. A plan that
passes `/r-plan`'s Step 3.0 will pass `/r-build`'s Section 1 — discrepancy
is a bug in this document, not in the caller.

## Rule 8: Tests-may-touch (optional)

Phases that add or extend tests MAY declare a scope-flex zone of paths
that the engineer is pre-authorized to touch without surfacing as an
out-of-scope finding:

```
**Tests-may-touch:** tests/fixtures/*.json, tests/helpers/*.bash
```

### 8a. Syntax

Comma-separated list of glob expressions or directory prefixes. Globs
use shell-style `*` (matches a single path component, no `/`). Examples:
- `tests/fixtures/*.json` — any JSON file directly in `tests/fixtures/`
- `tests/helpers/` — any file under `tests/helpers/` (recursive)
- `tests/**` — rejected (recursive `**` is too permissive; use a
  specific subdirectory)

### 8b. Ceilings

Drift inside the flex zone is bounded:

- **Per-file ceiling:** ≤30 lines added/changed (counted via
  `git diff --cached --numstat | awk '{print $1+$2}'`)
- **Global ceiling:** ≤3 files touched per phase in the flex zone

Drift exceeding either ceiling is rejected — same handling as
out-of-scope drift. The intent is to legitimize *trivial* test-infra
additions (a fixture, a helper); substantive test rewrites still
require explicit `**Files:**` declaration.

### 8c. Default

Empty (no flex zone). Phases without the `**Tests-may-touch:**` field
get strict `**Files:**`-only enforcement.

### 8d. Enforcement

Three call sites consume this rule:

- **Pre-commit hook** (`state/git-hooks/pre-commit`, installed in
  worktrees by dispatcher): primary gate; rejects `git commit` if
  staged files violate the union of `**Files:**` and `**Tests-may-touch:**`,
  or if either ceiling is exceeded.
- **Dispatcher post-merge check** (defense-in-depth): runs
  `git diff-tree --name-only` after engineer returns; same union check.
- **Engineer dirty check** (`canonical/agents/engineer.md` Setup):
  runs `git status --porcelain` before any aggregation/build step;
  same union check.

All three derive scope via `rdf_parse_phase_scope` from
`state/rdf-bus.sh`.

**Failure (out of scope):** *"Phase N: file <path> not in **Files:**
list and not matched by **Tests-may-touch:** glob."*

**Failure (per-file ceiling):** *"Phase N: file <path> in flex zone
changed <K> lines (ceiling 30)."*

**Failure (global ceiling):** *"Phase N: <K> files in flex zone
(ceiling 3)."*

## Adding a New Rule

When extending this schema:

1. Edit this file — add the rule with allowed values, failure message,
   and the call sites that enforce it.
2. Bump the `**Plan Version:** X.Y.Z` floor in the Plan-Version Awareness
   section if the rule is **stricter** than the prior version (existing
   plans may not satisfy it).
3. No edits to `/r-plan`, `/r-build`, or `reviewer` are required — they
   already cite this file by reference.
