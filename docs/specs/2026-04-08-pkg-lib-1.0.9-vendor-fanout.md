# pkg_lib v1.0.9 Release + Cross-Project Vendor Fanout

**Date:** 2026-04-08
**Phase:** 0 (Action A from `HANDOFF-packaging-hardening.md`)
**Targets:** pkg_lib (`master`), BFD PR #7 (`2.0.2`), LMD PR #478 (`2.0.1`)
**Disposition:** **In-tree to open 2.0.x PRs.** Time-sensitive.
**Umbrella:** `2026-04-08-packaging-hardening-strategy.md`
**Plan:** `rdf/docs/plans/2026-04-08-pkg-lib-1.0.9-vendor-fanout.md`

## Problem

`pkg_lib` has a staged but uncommitted v1.0.9 fix on `master`:

- `files/pkg_lib.sh` — version bump 1.0.8 → 1.0.9, new `is_assignment()` anchor regex, refactored `pkg_config_merge()` to distinguish shell assignments from conditional expressions
- `tests/09-config.bats` — regression test (bash conditional passthrough + `bash -n` parse validation)
- `CHANGELOG`, `CHANGELOG.RELEASE` — release notes for v1.0.9

Pre-1.0.9 `pkg_config_merge` treated any line containing `=` as `VAR=value`, corrupting conditional expressions like `[ "$X" = "1" ]` and `[[ "$Y" == "auto" ]]` into `[ "$X"= "1" ]` / `[[ "$Y"== "auto" ]]` by stripping the space before the operator. In APF, this surfaced as a class of CI failures where `importconf` merged `internals.conf` on every reinstall — landed in APF as `b0e54f0`.

**Current vendor state:**

| Project | `files/internals/pkg_lib.sh` version | Status |
|---|---|---|
| APF `2.0.2` | **1.0.9** | in-sync with master fix |
| BFD `2.0.2` | 1.0.8 | **behind** — ships corruption bug |
| LMD `2.0.1` | 1.0.8 | **behind** — ships corruption bug |

BFD PR #7 and LMD PR #478 currently ship pkg_lib 1.0.8 and will regress any consumer with conditional-expression config merges. This is a bug fix; deferral ships broken bits.

## Goal

1. Land the staged pkg_lib dirty state as a proper v1.0.9 release (commit + tag).
2. Vendor-sync BFD PR #7 to pkg_lib 1.0.9 in a single commit on the `2.0.2` branch.
3. Vendor-sync LMD PR #478 to pkg_lib 1.0.9 in a single commit on the `2.0.1` branch.
4. Keep APF #52 untouched — already at 1.0.9.

## Non-goals

- Do **not** fold Phase 3's manifest primitive into v1.0.9 (would delay urgent fix — explicitly rejected in strategy doc).
- Do **not** refactor `pkg_config_merge` beyond the minimum needed to pass the new regression test.
- Do **not** bundle any other BFD/LMD changes into the vendor-sync commits. One-file delta per consumer commit.
- Do **not** touch APF #52.
- Do **not** retag or rewrite history on any of the three branches.

## Design

### pkg_lib side (release)

pkg_lib's working tree already contains the fix. Release work is purely mechanical:

1. Verify the regression test passes locally (`make -C tests test` against Debian 12 minimum — see CLAUDE.md §Testing).
2. Stage exactly: `files/pkg_lib.sh`, `tests/09-config.bats`, `CHANGELOG`, `CHANGELOG.RELEASE`. **Do not stage** `SECURITY-FINDINGS.md` (untracked working file) or `docs/` (excluded).
3. Commit with body:
   ```
   v1.0.9 | pkg_config_merge: anchor variable detection to shell assignment regex

   [Fix] pkg_config_merge: anchor variable detection to ^[[:space:]]*[A-Za-z_][A-Za-z0-9_]*=
   [New] Regression test: tests/09-config.bats bash conditional passthrough + bash -n parse check
   [Change] Function header documents merge remains unsafe for conditional assignments (branches collapse)
   ```
4. Tag: `git tag -a v1.0.9 -m "pkg_lib v1.0.9"`.
5. Push `master` and tag to origin.

### BFD side (vendor sync)

Single-file update to `brute-force-detection/files/internals/pkg_lib.sh`:

1. Check out from the released pkg_lib v1.0.9 tag (not master HEAD — tag is the stable anchor).
2. Copy into BFD at the same path.
3. Verify BFD's own test suite still passes. BFD does not have `tests/09-config.bats`, so the smoke validation is:
   - Existing BFD install/upgrade tests stay green (anvil Debian 12 + Rocky 9 minimum)
   - Optional: add a single smoke test that calls `pkg_config_merge` on a fixture with conditional expressions. **Defer this** — the test already lives in pkg_lib and proves the primitive. BFD's test suite only needs to prove integration still works.
4. Commit to `2.0.2` branch with body:
   ```
   2.0.2 | Vendored pkg_lib 1.0.8 → 1.0.9: pkg_config_merge multi-var fix

   [Fix] Sync files/internals/pkg_lib.sh to pkg_lib v1.0.9
   [Fix] pkg_config_merge no longer corrupts conditional expressions in importconf
   ```
5. Push to PR #7 branch.

### LMD side (vendor sync)

Identical pattern to BFD. Target: `linux-malware-detect/files/internals/pkg_lib.sh`.

Commit message format differs (LMD uses `[Type] description` not `VERSION | description`):
```
[Fix] Vendored pkg_lib 1.0.8 → 1.0.9: pkg_config_merge multi-var fix

[Fix] Sync files/internals/pkg_lib.sh to pkg_lib v1.0.9
[Fix] pkg_config_merge no longer corrupts conditional expressions in importconf
```

Push to PR #478 branch.

## Sequencing

Strict ordering — each step gates the next:

1. pkg_lib v1.0.9 committed + tagged + pushed
2. BFD vendor sync picked from the tag, committed, pushed
3. LMD vendor sync picked from the tag, committed, pushed

Steps 2 and 3 could run in parallel worktrees since they touch independent repositories, but the in-tree commit is a single-file update and the serialization overhead is negligible. **Recommend sequential execution** to simplify rollback if CI reveals a regression.

### Regression guard

Before each vendor-sync push, confirm:

- The consumer's `importconf` path is exercised by at least one existing test. If not, note in the commit that integration-level coverage is deferred to Phase 1 migration fixtures.
- No unrelated changes sneak in — `git diff --stat HEAD^` should show exactly one modified file: `files/internals/pkg_lib.sh`.

## Risks

| Risk | Mitigation |
|---|---|
| pkg_lib regression test fails under real shells (unexpected portability issue) | Run tests on Debian 12 **and** Rocky 9 before tagging; bash 4.1 floor per CLAUDE.md |
| Vendor sync introduces an unrelated whitespace or line-ending drift | Use `git -C pkg_lib show v1.0.9:files/pkg_lib.sh > <consumer>/files/internals/pkg_lib.sh` to guarantee byte-for-byte parity |
| BFD or LMD has a private hot-fix in its vendored copy that differs from pkg_lib master | Audit `git log --oneline files/internals/pkg_lib.sh` in each consumer before overwriting — if any diff is a local hotfix, escalate before vendoring |
| Consumer CI discovers an unknown incompatibility with 1.0.9 | Abort the consumer commit, investigate, file as a pkg_lib follow-up before retrying |
| PR #7 or #478 is force-pushed after the vendor commit by another contributor | Low probability on these solo branches; mitigate by merging the vendor commit promptly after push |
| Commit message format wrong (missing `VERSION |` prefix on BFD, wrong bracket type on LMD) | Spec enumerates exact commit message format per consumer — QA gate must diff against this |

## Acceptance criteria

1. pkg_lib `master` HEAD has `v1.0.9` tag pointing at a clean release commit
2. BFD `2.0.2` branch HEAD has exactly one new commit: vendor sync of `files/internals/pkg_lib.sh` to 1.0.9
3. LMD `2.0.1` branch HEAD has exactly one new commit: vendor sync of `files/internals/pkg_lib.sh` to 1.0.9
4. All three pushes complete without CI regression
5. APF PR #52 is **not** touched

## Out of scope

- Any post-commit validation that `importconf` behaves correctly in a live-upgrade scenario — covered by Phase 1 migration fixtures
- Any changes to `pkg_config_merge` semantics beyond the 1.0.9 fix
- Release notes coordination with APF (APF already has the fix in-tree; no cross-project release note is needed)
- BFD/LMD CHANGELOG.RELEASE updates beyond the single `[Fix]` line in the commit — the commit itself is the release note for the vendor sync
