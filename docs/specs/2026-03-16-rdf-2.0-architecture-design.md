# RDF 2.0 Architecture Design Spec

> Produced from brainstorming session 2026-03-16.
> Inputs: Claude Code usage report (273 sessions, 14 days), current live
> tooling state (`/root/.claude/`), workforce repo state, community best
> practices research, RDF 1.0 architecture (RDF.md + PLAN.md).

---

## 1. Design Decisions

### 1.1 RDF Is Authoritative

RDF is the single source of truth for all agent definitions, commands,
scripts, and governance that get deployed into Claude Code, Gemini CLI,
or Codex environments.

```
PREVIOUS (replaced):
  Develop in ~/.claude/ -> sync.sh -> workforce/claude/ -> commit
  (live is source of truth, repo is backup)

NEW:
  Develop in rdf/canonical/ -> rdf generate -> deploy to /root/.claude/
  (rdf/ is source of truth, /root/.claude/ is deployment target)
```

- `/root/.claude/{commands,agents,scripts}` become symlinks to
  `rdf/adapters/claude-code/output/`
- `sync.sh` and `install.sh` replaced by `rdf generate` + `rdf sync`
- Development happens in `rdf/canonical/`
- Emergency escape hatch: edit `/root/.claude/` directly, then
  `rdf sync` pulls changes back to canonical (strips frontmatter)
- `rdf doctor` detects drift between canonical and generated output

Parent CLAUDE.md rule changes to:
> All agent, command, and script development happens in rdf/canonical/.
> Run `rdf generate claude-code` to deploy changes to /root/.claude/.
> Never edit /root/.claude/ directly except for emergency fixes
> (follow up with `rdf sync` to pull changes back to canonical).

### 1.2 Naming Convention

**Pattern:** `{domain}-{role}` for domain-specific, `{role}` for core.

Three tiers:

| Tier | Prefix | Agents | Rationale |
|------|--------|--------|-----------|
| Core | none | mgr, po, scope | Orchestrate across any domain |
| Domain | {domain}- | sys-eng, sys-qa, sys-uat, sys-sentinel, sys-challenger, sys-ux | Domain expertise in agent prompt |
| Specialist | {domain}- | sec-eng, fe-qa, fe-uat | Cross-cutting or domain-scoped |

**Domain registry:**

| Shortcode | RDF Profile | Domain |
|-----------|-------------|--------|
| sys | systems-engineering | Bash/shell, Linux, security tooling |
| sec | security | Offensive/defensive security assessment |
| fe | frontend | Vue/React, CSS, Playwright |
| php | php-backend | PHP, MySQL, cPanel/Plesk apps (future) |
| py | python-backend | Python/Perl, Postgres (future) |
| iaas | infrastructure | IaaS, API integrations, cloud (future) |
| fs | full-stack | Cross-stack coordination (future) |

**CC agent names:** `rfxn-{file-stem}` (e.g., `rfxn-sys-eng`, `rfxn-mgr`)
**Slash commands:** `/{file-stem}` (e.g., `/sys-eng`, `/mgr`)

### 1.3 Superpowers Boundary

RDF is self-sufficient. No dependency on the superpowers plugin.

- **RDF owns:** the rfxn engineering pipeline (agents, audit, release,
  tiered verification, adversarial review)
- **Superpowers owns:** generic dev methodology (brainstorming, TDD,
  systematic debugging)
- **No coupling:** if superpowers is installed, it handles generic
  workflow before RDF's pipeline. If not, RDF still works.
- **Pattern absorption:** best ideas from superpowers (scope
  confirmation, self-verification, evidence-before-assertions) are
  baked into RDF agent definitions during canonical migration.

### 1.4 Overwatch Status

Overwatch project is dead. All Overwatch-specific artifacts removed:
- `overwatch-hook.sh` script — removed
- Overwatch-specific content stripped from fe-qa and fe-uat agents

Frontend profile (`fe-*` agents) preserved as generic web/frontend
personas ready for whatever web framework comes next.

### 1.5 GitHub-Native Project Management

GitHub Issues + Projects v2 is the durable work tracking layer.
Standardized across all rfxn repos.

- Phase issues (`type:phase`) contain detailed designs
- Task issues (`type:task`) are pullable work units
- Milestones map to phases or releases
- Org-level Ecosystem Project provides cross-repo visibility
- Repo-level Development Project provides per-project kanban + roadmap

PLAN.md and MEMORY.md remain as session-local agent context but GitHub
Issues is the source of truth for queue state.

## 4. CLI Design

Single `rdf` dispatcher with lazy-sourced subcommand modules.
Mirrors the Sigforge `sf` pattern: two-level case dispatch,
help-before-init, idempotent `rdf_init()`, portable binary discovery.

### Usage

```
Usage: rdf <command> [subcommand] [options]

RDF 2.0.0 -- rfxn Development Framework

Commands:
  generate   Build tool-specific files from canonical sources
  profile    Manage active domain profiles
  init       Initialize projects with RDF conventions
  doctor     Check project health and convention drift
  state      Deterministic project state snapshot (JSON)
  refresh    Agent-driven state file updates
  sync       Pull /root/.claude/ changes back to canonical
  github     GitHub Issues + Projects integration

Options:
  help       Show this help
  version    Show version

Run 'rdf <command> help' for subcommand details.
```

### Subcommand Reference

```
rdf generate claude-code              # canonical + profiles -> CC output
rdf generate gemini-cli               # canonical + profiles -> Gemini config
rdf generate codex                    # canonical + profiles -> Codex config
rdf generate agents-md                # canonical + profiles -> AGENTS.md
rdf generate all                      # all active adapters

rdf profile list                      # show profiles with deps
rdf profile install <name> [<name>...]# activate + resolve deps + regenerate
rdf profile remove <name>             # deactivate, warn if dependents active
rdf profile status                    # active profiles + component counts

rdf init <path> [options]             # initialize project
  --type shell|lib|frontend|security|minimal
  --tools claude-code,gemini-cli,codex
  --version X.Y.Z
  --no-memory
  --github                            # create labels + repo project board
  --batch                             # process multiple directories

rdf doctor [<path>] [options]         # health check
  --all                               # all workspace projects
  --scope artifacts|drift|memory|plan|github|sync

rdf state [<path>]                    # project -> JSON to stdout (<1s)

rdf refresh [<path>] [options]        # agent-driven state updates
  --scope memory|plan|github|all

rdf sync                              # /root/.claude/ -> canonical (reverse)

rdf github setup [--repo <owner/repo>]       # labels + repo project
rdf github sync-labels [--org <org>]         # sync taxonomy across repos
rdf github ecosystem-init [--org <org>]      # org-level project
rdf github ecosystem-add <owner/repo>        # add repo to ecosystem project
```

### Dispatcher Pattern

```bash
#!/usr/bin/env bash
set -euo pipefail

RDF_HOME="$(cd "$(dirname "$0")/.." && pwd)" || exit 1
RDF_LIBDIR="${RDF_HOME}/lib"

source "${RDF_LIBDIR}/rdf_common.sh"

case "${1:-}" in
    generate) shift; source "${RDF_LIBDIR}/cmd/generate.sh"; cmd_generate "$@" ;;
    profile)  shift; source "${RDF_LIBDIR}/cmd/profile.sh";  cmd_profile "$@" ;;
    init)     shift; source "${RDF_LIBDIR}/cmd/init.sh";     cmd_init "$@" ;;
    doctor)   shift; source "${RDF_LIBDIR}/cmd/doctor.sh";   cmd_doctor "$@" ;;
    state)    shift; source "${RDF_LIBDIR}/cmd/state.sh";    cmd_state "$@" ;;
    refresh)  shift; source "${RDF_LIBDIR}/cmd/refresh.sh";  cmd_refresh "$@" ;;
    sync)     shift; source "${RDF_LIBDIR}/cmd/sync.sh";     cmd_sync "$@" ;;
    github)   shift; source "${RDF_LIBDIR}/cmd/github.sh";   cmd_github "$@" ;;
    help|--help|-h|"") usage ;;
    version|--version|-V) echo "rdf ${RDF_VERSION}" ;;
    *)        rdf_die "unknown command: $1 -- run 'rdf help' for usage" ;;
esac
```

### 5.3 Workflow States

All Development Projects use this 5-state workflow on the Status field:

| State | Color | Description | Transition From | Transition To |
|-------|-------|-------------|----------------|---------------|
| Backlog | Gray | Not yet scheduled | — | Ready |
| Ready | Blue | Dependencies met, ready to pull | Backlog | In Progress |
| In Progress | Yellow | Actively being worked | Ready | In Review |
| In Review | Orange | QA/UAT/peer review gate | In Progress | Done, In Progress (rework) |
| Done | Green | Complete and verified | In Review | — |

When working issues, agents MUST transition status:
- `/sys-eng` picks issue → set **In Progress**
- `/sys-eng` commits + reports → set **In Review**
- `/sys-qa` approves → set **Done**
- `/sys-qa` requests changes → set **In Progress** (rework)

### 5.4 Project Views

Every Development Project board MUST have a **Kanban** view as the
default Board layout. This is the primary working view.

`rdf github setup` creates API-managed fields and items, then instructs
the user to create the Kanban view (Board layout creation requires the
GitHub web UI — not available via API).

**Repo-level Development Project (5 views):**

| View | Layout | Column Field | Purpose |
|------|--------|-------------|---------|
| **Kanban** | Board | Status | **DEFAULT** — primary working view, 5 columns |
| Phase Board | Board | Phase | Kanban grouped by phase |
| Active Work | Board | Status, filtered In Progress + In Review | Current work only |
| Roadmap | Roadmap | Phase | Timeline view |
| Backlog | Table | filtered Backlog + Ready, sorted by Priority | Pull queue |

**Org-level Ecosystem Project (4 views):**

| View | Layout | Column Field | Purpose |
|------|--------|-------------|---------|

| Cross-Project Board | Board | Project | Work grouped by project |
| Project Roadmap | Roadmap | Project | Release timelines per project |

### 5.6 Workflow Integration

The same pattern applies to all rfxn projects:

```
/mgr       -> reads open issues from repo project board, prioritizes
/sys-eng   -> picks issue, implements, references issue # in commit
/sys-qa    -> reviews, comments findings on issue
/sys-sentinel -> post-impl findings as issue comments
Merge      -> issue closed, Status field -> Done
/audit     -> findings become type:audit-finding issues
Release    -> release-gate label filters blockers
```

`rdf init --github` scaffolds this for any new project.
`rdf github sync-labels` keeps taxonomy consistent across repos.

## 7. Friction Fixes

Fixes 1-3 injected into Phase 1. Fix 4 spans Phases 2 + 4.

Based on usage report analysis (273 sessions, 14 days):

| Friction | Root Cause | Fix | Target |
|----------|-----------|-----|--------|
| False-positive QA findings (wrong line numbers) | QA/Sentinel cite without verifying | Self-verification gate: before presenting findings, verify each file:line contains described issue | sys-qa.md, sys-sentinel.md |
| Scope misalignment (wrong-scope plans) | EM dispatches SE without scope confirmation | Scope confirmation protocol: 3 bullets before SE dispatch, wait for user approval | mgr.md |
| Unverified evidence claims | SE claims "done" without proof | Strengthen Step 7: grep evidence for bash 4.1, refactor grep evidence mandatory, not optional | sys-eng.md |
| Cross-project context drift | Session starts in wrong repo | Context anchoring: /reload runs rdf state, displays project/branch/version/dirty | reload command, rdf-state.sh (Phase 2 + Phase 4, not Phase 1) |

Key insight: RDF independently converged on patterns the community is
formalizing (typed agents, file-based coordination, context window
management, convention governance). The community has not reached RDF's
depth on adversarial review gates, tiered verification, or cross-project
convention inheritance.
