# RDF 3.0 Architecture Design

**Date:** 2026-03-18
**Status:** Draft
**Author:** Ryan MacDonald / RDF Planner

---

## Summary

RDF 3.0 is a ground-up rationalization of the rfxn Development Framework.
The framework shifts from domain-specific agent personas to universal agent
primitives that derive domain knowledge from a data layer. Governance is
generated from project codebases rather than maintained as static profile
documents. The command surface collapses from ~68 slash commands to 23 (13 lifecycle
+ 10 utility), all namespaced under `/r:` to avoid collision with Claude
Code built-ins.

**The 30-second pitch:**

> RDF gives you 6 universal agents, 23 commands, and automatic governance
> generation. Point it at any codebase and start building. It researches
> best practices, challenges your assumptions, writes specs, executes plans
> with TDD, and runs adversarial quality gates. No configuration. No domain
> taxonomy. Just `/r:init` and go.

## Design Principles

### 1. Agents Define Role and Protocol, Data Provides Domain Knowledge

Agent prompts are domain-agnostic. They define what the role does (engineer,
QA, reviewer) and the protocol it follows (TDD, verification gates, adversarial
passes). Domain-specific knowledge — coding conventions, platform constraints,
anti-patterns — comes from the project's governance data layer.

The same engineer agent becomes a bash specialist in an rfxn project and a
React specialist in a frontend project because the governance data tells it
to be one.

### 2. Governance Is Generated, Not Authored

Instead of maintaining static governance profiles per domain, RDF provides
initialization primitives that analyze a project's codebase and generate
governance collateral automatically. Existing convention files (CLAUDE.md,
AGENTS.md, GEMINI.md) are ingested as highest-priority inputs. Codebase
scanning fills gaps.

### 3. Supplement, Never Replace

RDF generates a supplementary governance layer alongside existing project
files. Existing CLAUDE.md, AGENTS.md, and other convention files remain
authoritative and untouched. RDF adds value without taking over.

### 4. Progressive Disclosure for Context

Only a lightweight governance index (~50 lines / ~100-150 tokens) is always loaded.
Agents read the index, decide what they need for the current task, and
load specific governance files just-in-time. This follows Anthropic's
context engineering guidance: "just-in-time context over pre-loading
everything."

### 5. Project Governance vs. Operational Mode

Project governance is intrinsic — how the project is built. Operational
modes are extrinsic — what the operator is doing right now (development,
security assessment, performance audit). These are separate axes that
compose independently.

## Section 1: Agent Architecture

### 6 Universal Agents

| Agent | Type | Model | Purpose |
|-------|------|-------|---------|
| **planner** | Skill (main context) | opus | Brainstorm, scope, spec, implementation plan. Research-driven adversarial collaboration with user. Interactive — requires back-and-forth with user. |
| **dispatcher** | Orchestrator | sonnet | Execute plans, manage quality gates, TDD red/green cycles, dispatch subagents. Three execution modes. |
| **engineer** | Subagent (execution) | opus | Implementation in any domain. Reads governance for conventions and constraints. |
| **qa** | Subagent (read-only) | sonnet | Verification gate. Reads governance for domain-specific checks, lint, tests. |
| **uat** | Subagent (read-only) | sonnet | Acceptance testing. Real-world scenarios against installed artifacts. |
| **reviewer** | Subagent (read-only) | opus | Adversarial review with two modes: **challenge** (pre-impl) and **sentinel** (post-impl). |

### Agent Responsibilities

**Planner** operates in the main conversation context. It is interactive
and collaborative — it needs back-and-forth with the user. It researches
industry best practices, challenges the user's assumptions with evidence,
and always presents at minimum three options for every architectural
decision. It produces specs and plans but never writes code.

**Dispatcher** takes a finished plan and executes it phase by phase. It
reads the execution mode tag on each phase, dispatches appropriate
subagents, manages TDD cycles, enforces quality gates, and synthesizes
results. It decides which quality gates apply based on phase complexity
and risk tags set during planning.

**Engineer** is a focused implementation agent. It receives a phase
description, file ownership boundaries, and governance context. It follows
TDD: write test, red, implement, green, refactor. It returns structured
evidence of its work (test names, red/green output, coverage delta).

**QA** is a read-only verification agent. It reads the project's
verification.md governance file to know which checks to run (lint, type
checks, anti-pattern greps, test execution). It produces a structured
pass/fail report.

**UAT** is a read-only acceptance testing agent. It runs real-world
scenarios — install flows, CLI interactions, multi-step workflows, failure
recovery. It operates from a sysadmin/end-user persona.

**Reviewer** is a read-only adversarial review agent with two modes:

- **Challenge mode** (pre-implementation): Reviews specs and plans for
  design flaws, missed edge cases, simpler alternatives, and
  architectural risks. Dispatched during the planner's spec phase.
- **Sentinel mode** (post-implementation): Reviews diffs for anti-slop
  (unnecessary changes), regressions, security vulnerabilities, and
  performance issues. Dispatched during the dispatcher's quality gates.

### What Disappeared from RDF 2.x

| RDF 2.x Agent | RDF 3.0 Equivalent |
|---|---|
| mgr | Absorbed by dispatcher |
| po | Absorbed by planner |
| scope | Absorbed by planner (discover phase) |
| sys-eng, fe-eng | engineer (governance provides domain knowledge) |
| sys-qa, fe-qa | qa (governance provides domain-specific checks) |
| sys-uat, fe-uat | uat (governance provides domain scenarios) |
| sys-challenger | reviewer --challenge |
| sys-sentinel | reviewer --sentinel |
| sys-ux | Absorbed by qa (governance defines UX checks when relevant) |
| sec-eng | reviewer in security mode, or engineer with security governance |

**12 agents down to 6. Zero domain prefixes.**

Note: RDF 2.x had asymmetric coverage — the frontend profile had `fe-qa`
and `fe-uat` but no `fe-eng` or `fe-ux`. The systems-engineering profile
had the most complete agent set. With universal agents, this asymmetry
disappears — every project gets the full set of 6 regardless of domain.

## Section 2: Planner Workflow

The planner is invoked via `/r:plan` and operates in the main conversation
context.

```
/r:plan
  |
  +-- PHASE 1: DISCOVER
  |   Read governance index -> load architecture.md
  |   Read existing PLAN.md / GitHub issues if present
  |   Assess scope: single spec or decompose into sub-projects?
  |
  +-- PHASE 2: BRAINSTORM + RESEARCH
  |   For each design question:
  |     1. Research industry best practices (web search, codebase patterns,
  |        framework docs, prior art) BEFORE forming options
  |     2. Present 3 options with trade-offs, cite sources
  |     3. Make a recommendation -- but challenge the user's assumptions
  |        if research suggests a better path
  |     4. User selects or proposes alternative
  |
  |   Adversarial posture:
  |     - "You asked for X, but the industry has moved to Y because..."
  |     - "This is a solved problem -- library Z handles this, here's the
  |        trade-off vs building it yourself..."
  |     - "Your approach works, but here's an edge case at scale..."
  |     - Present user's preferred path as one of the 3 options, never dismiss
  |
  |   Research scope per question:
  |     - Use Claude Code's built-in Explore subagent for codebase pattern
  |       discovery (parallel, background) -- this is a CC native primitive,
  |       not an RDF agent
  |     - Web search for current best practices, framework recommendations
  |     - Check existing governance/reference docs for prior decisions
  |
  |   One question at a time. Multiple choice when possible.
  |   Every option is research-backed, not just inference.
  |
  +-- PHASE 3: SPEC
  |   Write design doc to docs/specs/YYYY-MM-DD-<topic>-design.md
  |   Dispatch reviewer in challenge mode (pre-impl review of spec)
  |   Fix issues, re-dispatch (max 3 review-fix cycles, then surface
  |   unresolved issues to human for guidance)
  |   User reviews and approves written spec
  |
  +-- PHASE 4: PLAN
  |   Decompose spec into numbered phases
  |   For each phase: files touched, acceptance criteria, test strategy
  |   Tag phases with execution mode:
  |     [serial-context]  -- small, single-file, stays in main session
  |     [serial-agent]    -- medium, multi-file, one subagent at a time
  |     [parallel-agent]  -- large, independent file sets, parallel subagents
  |   Identify file ownership boundaries for parallel phases
  |   Write plan to PLAN.md
  |   User reviews and approves plan
  |
  +-- HANDOFF
      Plan is ready for /r:build (dispatcher takes over)
```

### Key Design Decisions

- **Planner never writes code** — it produces specs and plans only.
- **Challenge review happens on the spec**, not after implementation.
  Shift-left: catch design flaws before writing code.
- **Each plan phase is tagged with an execution mode** so the dispatcher
  knows how to run it.
- **File ownership boundaries are explicit** in the plan, preventing
  parallel conflicts before they happen.
- **Research before opinions** — the planner does homework (web search,
  codebase exploration, prior art) before presenting options. Every
  architectural decision gets at minimum three research-backed options.
- **Collaborative adversary** — the planner pushes back with evidence when
  the user's instinct conflicts with best practices, but always presents
  the user's preferred direction as a valid option.

## Section 3: Dispatcher & Execution Modes

The dispatcher is invoked via `/r:build [phase-N]` and manages the full
execution lifecycle.

```
/r:build [phase-N]
  |
  +-- LOAD
  |   Read PLAN.md -> identify target phase
  |   Read governance index -> load relevant context for phase
  |   Determine execution mode from phase tag
  |
  +-- EXECUTE (one of three modes)
  |
  |   [serial-context] -- small changes, main session
  |   |  Dispatcher stays in conversation context
  |   |  TDD cycle: write test -> red -> implement -> green -> refactor
  |   |  Verify in-line, no subagent overhead
  |   |  Best for: single-file fixes, config changes, doc updates
  |   |
  |   [serial-agent] -- medium changes, one subagent at a time
  |   |  Dispatcher spawns engineer subagent with phase context
  |   |  Engineer follows TDD: write test -> red -> implement -> green
  |   |  Engineer returns result -> dispatcher runs qa subagent
  |   |  QA returns result -> dispatcher evaluates gate
  |   |  Best for: multi-file changes with dependencies between files
  |   |
  |   [parallel-agent] -- large changes, parallel subagents
  |      Dispatcher validates file ownership boundaries (no overlapping files)
  |      Spawns N engineer subagents, each with isolated file set
  |      Each engineer follows TDD independently
  |      Gate: all engineers complete -> dispatcher runs qa across full diff
  |      Conflict detection: if an engineer needs a file outside its boundary,
  |        it stops and reports back -- dispatcher reassigns or serializes
  |      Best for: independent modules, parallel test suites, multi-component
  |
  +-- QUALITY GATES (after each phase)
  |
  |   Gate 1: Engineer self-report
  |     TDD evidence: test names, red/green output, coverage delta
  |
  |   Gate 2: QA verification (subagent)
  |     Reads governance/verification.md for project-specific checks
  |     Runs lint, type checks, anti-pattern greps, test execution
  |     Produces structured pass/fail report
  |
  |   Gate 3: Reviewer (conditional, for significant changes)
  |     Sentinel mode: anti-slop, regression, security, performance
  |     Dispatcher decides based on phase complexity/risk tag in plan
  |
  |   Gate 4: UAT (conditional, for user-facing changes)
  |     Real-world scenarios, install flows, CLI interactions
  |     Dispatcher decides based on phase type tag in plan
  |
  |   RED/GREEN DECISION
  |     All gates pass -> phase complete, update PLAN.md, next phase
  |     Any gate fails -> engineer gets feedback, re-enters TDD cycle
  |     Max 3 retry loops -> surface to user with failure context
  |
  +-- STATUS OUTPUT
  |   After each phase: structured status update to work-output/
  |   /r:status reads these for project health dashboard
  |
  +-- COMPLETION
      All phases done -> /r:ship available for release workflow
```

### Execution Mode Selection

Execution modes are tagged at planning time by the planner, not decided
at runtime by the dispatcher. The planner tags each phase based on:

- **File count**: 1 file = serial-context, 2-5 files = serial-agent,
  6+ independent files = parallel-agent
- **File dependencies**: shared files force serial execution
- **Risk level**: high-risk changes (security, data migration) prefer
  serial for tighter gate control

The dispatcher respects the plan tags but can downgrade (parallel to serial)
if it detects file boundary violations at runtime. It never upgrades
(serial to parallel) without replanning.

### TDD Protocol

TDD is the default for all execution modes:

1. **Red**: Write a failing test that defines the acceptance criteria
2. **Green**: Write the minimum implementation to pass the test
3. **Refactor**: Clean up while keeping tests green
4. Evidence of each step is captured and passed to quality gates

### Parallel Failure Semantics

When an engineer fails in parallel mode:

1. **Running engineers are NOT interrupted** — they continue to completion.
   Their work is valid and should not be discarded.
2. **Failed engineer enters retry** — the dispatcher re-dispatches it with
   failure context. Other completed engineers' outputs are preserved.
3. **Retry is per-track, not per-phase** — only the failed track retries.
   The 3-retry limit applies per individual engineer, not across the phase.
4. **Integration step** — after all engineers complete (including retries),
   the dispatcher runs an integration check before the QA gate. This
   catches semantic conflicts between independently-written changes
   (e.g., two engineers adding imports to a shared test file).
5. **If integration fails** — the dispatcher serializes the conflicting
   tracks and re-runs them sequentially with shared context.
6. **If 3 retries exhausted on any track** — the phase surfaces to the
   user with: which tracks succeeded, which failed, and failure context.

### Commit Strategy

Commits follow the project's existing governance (from CLAUDE.md or
governance/conventions.md). The default protocol:

- **Serial-context**: dispatcher commits after all quality gates pass.
  One commit per phase.
- **Serial-agent**: engineer produces changes, dispatcher commits after
  QA gate passes. One commit per phase.
- **Parallel-agent**: each engineer works in a git worktree. After all
  engineers complete and integration passes, the dispatcher merges
  worktrees and creates a single commit per phase. This avoids
  intermediate commits from individual engineers.
- **Commit messages**: generated from the phase description and acceptance
  criteria in PLAN.md, following the project's commit convention
  (detected during governance init from git log patterns).
- **The dispatcher always commits**, never individual engineer subagents.
  This ensures one commit per logical unit regardless of execution mode.

### Skip-Planner Path

`/r:build` does not require `/r:plan` to have been run. It requires only
a PLAN.md file with the minimum schema:

```markdown
## Phase N: <description>
- **Mode**: serial-context | serial-agent | parallel-agent
- **Files**: <file list or "all">
- **Accept**: <acceptance criteria>
- **Status**: pending | in-progress | complete
```

Users can write PLAN.md by hand, import it from an external tool, or
use `/r:plan` to generate it. The dispatcher validates the schema on
load and reports missing fields.

### Quality Gate Progressive Application

Not every phase needs all 4 gates. The plan tags each phase with risk
and type metadata. The dispatcher selects gates accordingly:

**Phase Tag Vocabulary:**

- **Risk levels**: `risk:low`, `risk:medium`, `risk:high`
- **Type values**: `type:config`, `type:feature`, `type:refactor`,
  `type:security`, `type:user-facing`, `type:data-migration`
- **Default** (when no tags are set): `risk:medium, type:feature`
  (Gates 1 + 2)

| Phase Tag | Gates Applied |
|---|---|
| `risk:low, type:config` | Gate 1 (self-report) only |
| `risk:medium, type:feature` | Gates 1 + 2 (self-report + QA) |
| `risk:high, type:security` | Gates 1 + 2 + 3 (+ reviewer sentinel) |
| `type:user-facing` | Gates 1 + 2 + 4 (+ UAT) |
| `risk:high, type:user-facing` | All 4 gates |

## Section 4: Governance Initialization

Invoked via `/r:init [path]`. Generates project-specific governance from
existing convention files and codebase analysis.

### Priority Cascade

```
1. CLAUDE.md / AGENTS.md / GEMINI.md     -- curated conventions (highest signal)
2. MEMORY.md / PLAN.md                   -- project state, decisions, context
3. .cursorrules / .cursor/rules/         -- Cursor conventions
4. .github/copilot-instructions.md       -- Copilot conventions
5. Codebase scan                         -- languages, frameworks, patterns
6. Tooling detection                     -- CI, linters, Dockerfiles, packages
7. Git history                           -- commit patterns, active areas
```

Each layer reinforces or fills gaps left by the layer above. If someone has
a detailed CLAUDE.md, the codebase scan validates and extends it. If there
is nothing, the scan builds everything from scratch.

### Init Flow

```
/r:init [path]
  |
  +-- PHASE 1: INGEST EXISTING .MD FILES (highest priority)
  |   Scan for convention files (see priority cascade above)
  |   Extract: conventions, constraints, architecture decisions, anti-patterns
  |   These files stay untouched -- they remain authoritative
  |
  +-- PHASE 2: CODEBASE SCAN
  |   Language detection (file extensions, shebangs, package manifests)
  |   Framework detection (package.json deps, imports, config files)
  |   Directory structure mapping (src/, tests/, lib/, etc.)
  |   Build system (Makefile, package.json scripts, CI configs)
  |   Test framework (jest, pytest, bats, go test, etc.)
  |   Linter/formatter configs (.eslintrc, .prettierrc, shellcheck, etc.)
  |
  +-- PHASE 3: TOOLING & INFRASTRUCTURE DETECTION
  |   CI/CD: .github/workflows/, .gitlab-ci.yml, Jenkinsfile
  |   Containers: Dockerfile, docker-compose.yml
  |   Platform targets: CI matrix, target OS in configs
  |   Dependencies: lockfiles, version constraints
  |   Git patterns: branch naming, commit conventions (from log)
  |
  +-- PHASE 4: GENERATE SUPPLEMENTARY GOVERNANCE
  |   Creates .claude/governance/ (never overwrites existing files)
  |
  |   index.md          -- file descriptions (~50 lines, always loaded)
  |   architecture.md   -- component map, data flow, key boundaries
  |   conventions.md    -- coding patterns, naming, style
  |   verification.md   -- what checks to run (from CI + linter + test framework)
  |   constraints.md    -- platform targets, compat requirements, version floors
  |   anti-patterns.md  -- project-specific pitfalls
  |
  |   Merge logic:
  |     - If existing .md files already cover a topic -> governance file
  |       references them ("see CLAUDE.md section Shell Standards")
  |       instead of duplicating
  |     - If no existing .md files -> governance file generates from scan
  |     - If partial coverage -> governance file supplements gaps only
  |
  +-- PHASE 5: VALIDATE
  |   Verify generated governance against codebase (spot-check accuracy)
  |   Flag low-confidence inferences for user review
  |   Report summary with confidence levels
  |
  +-- OUTPUT
      .claude/
        governance/
          index.md
          architecture.md
          conventions.md
          verification.md
          constraints.md
          anti-patterns.md

      Existing CLAUDE.md / AGENTS.md / etc. -- untouched
```

### Conflict Resolution

When sources disagree (e.g., CLAUDE.md says "use tabs" but .editorconfig
says "use spaces"):

1. **Higher-priority source wins** per the priority cascade.
2. **Conflicts are flagged** in the low-confidence report with both values
   shown, so the user can resolve.
3. **Within the same priority tier** (e.g., two linter configs disagree),
   the governance file documents both and flags for user resolution.
4. **The generated governance never silently picks a side** — it either
   follows the priority cascade or flags the conflict explicitly.

### Monorepo Handling

For monorepos with multiple languages/frameworks in subdirectories:

- `/r:init .` operates on the root and produces governance for the
  dominant patterns across the whole repo.
- `/r:init ./services/api` can scope to a subdirectory, producing
  governance specific to that service.
- When run at root, the architecture.md file documents the service
  boundaries and their individual technology stacks.
- Governance files reference subdirectory-specific conventions where
  they differ from the root (e.g., "services/api/ uses Python conventions;
  services/frontend/ uses TypeScript conventions").

### Output for Rich .md Projects

When existing convention files are comprehensive (like rfxn projects),
governance files are thin — mostly pointers back to existing sections
with supplementary gaps filled from the scan.

### Output for Zero .md Projects

When no convention files exist, phases 2-3 do all the heavy lifting.
Governance output is denser because there is nothing to reference. The
user gets a report showing everything was inferred, with low-confidence
items flagged for review.

### Directory Relationship

`.claude/governance/` is a subdirectory of Claude Code's existing `.claude/`
project directory. It does not conflict with `.claude/settings.json`,
`.claude/agents/`, or other CC-managed files. RDF only writes to the
`governance/` subdirectory and never modifies other `.claude/` contents.

For projects using RDF 2.x, the old symlink-based deployment
(`~/.claude/agents/` -> `rdf/output/agents/`) is replaced by the
governance generation model. RDF 3.0 agents live in `~/.claude/agents/`
as standard CC subagent files, not symlinks.

### Refresh

`/r:refresh` re-runs phases 2-5 against current codebase state, updating
governance files while preserving user modifications (tracked via a
`.claude/governance/.user-modified` marker).

## Section 5: Command Surface & Skill Architecture

### 13 Commands, All `/r:` Prefixed

The `r:` prefix avoids collision with Claude Code built-in commands
(`/init`, `/plan`, `/status`, `/review` are all taken). Outside
stakeholders learn one rule: RDF commands start with `/r:`.

```
COMMAND          WHAT HAPPENS                              CC PRIMITIVE
-------          ------------                              ------------
/r:init          Scan codebase, generate governance        Skill (main context)
/r:start         Load governance, display status anchor    Skill (main context)
/r:plan          Brainstorm -> spec -> implementation plan Skill (main context)
/r:build [N]     Invoke dispatcher agent                   Subagent (foreground)
/r:verify        Invoke qa subagent directly               Subagent (foreground)
/r:test          Invoke uat subagent directly               Subagent (foreground)
/r:review        Invoke reviewer subagent directly          Subagent (foreground)
/r:status        Project health dashboard                  Skill (main context)
/r:ship          Pre-release -> PR -> monitor              Skill + subagents
/r:mode [name]   Switch operational mode                   Skill (main context)
/r:refresh       Re-scan codebase, update governance       Skill (main context)
/r:audit         Full codebase audit                       Skill + subagents
/r:sync          Pull edits back to canonical              Skill (main context)
```

### Implementation Glossary

- **Skill (main context)**: A Claude Code skill `.md` file that runs
  inline in the user's conversation. Interactive — the user sees output
  and can respond. Implemented as files in `.claude/commands/` or
  `.claude/skills/`.
- **Subagent (foreground)**: A Claude Code subagent `.md` file
  (`.claude/agents/`) that runs in an isolated context window. Receives
  a task prompt, works independently, returns results. Cannot have
  interactive back-and-forth with the user during execution.
- **Skill + subagents**: A skill that runs in main context but dispatches
  subagents internally for parallel or isolated work (e.g., `/r:ship`
  runs pre-release checks inline, then dispatches qa/reviewer subagents
  for verification gates before creating a PR).

### Command-to-Primitive Mapping

- **7 skills** (run in main context, interactive): init, start, plan,
  status, mode, refresh, sync
- **3 direct subagent invocations** (manual control): verify, test, review
- **1 orchestrating subagent** (dispatches further subagents): build
- **1 orchestrating skill** (dispatches subagents): audit
- **1 composite workflow** (skill + subagents): ship

### Standalone Invocation Semantics

When `/r:verify`, `/r:test`, or `/r:review` are invoked directly (outside
of `/r:build`), they operate on the **current git diff** by default:

- `/r:verify` — runs QA checks against uncommitted changes
- `/r:test` — runs acceptance scenarios against the current working state
- `/r:review` — runs adversarial review against uncommitted changes
- `/r:review --challenge` — reviews a spec file (pass path as argument)
- `/r:review --sentinel` — reviews a diff (defaults to current branch diff)

All three accept an optional scope argument to target specific files
or directories.

### Utility Commands (/r:util:)

Utilities are on-demand maintenance tools that don't fit the lifecycle
workflow. They use a secondary namespace (`/r:util:`) and follow a
strict `{subject}-{verb}` naming convention.

**Naming rules:**
- Subject is a short noun grouping: `mem`, `doc`, `ci`, `lib`, `code`,
  `test`, `chg`
- Verb is an action: `compact`, `gen`, `scan`, `sync`, `dedup`,
  `release`, `modernize`
- New utilities must follow this pattern — no exceptions
- Subject-first enables autocomplete discovery (type `/r:util:lib` to
  see all library utilities)

```
COMMAND                    WHAT IT DOES                            SOURCE (2.x)
-------                    ------------                            ------------
/r:util:mem-compact        Archive stale MEMORY.md entries         mem-compact
/r:util:doc-gen            Generate docs for a specific surface    doc-author
/r:util:ci-gen             Generate CI workflow from test infra    ci-setup
/r:util:lib-sync           Cross-project library drift detection   proj-lib-sync
/r:util:lib-release        Shared library release lifecycle        lib-release
/r:util:code-scan          Structured pattern-class bug finder     code-grep
/r:util:code-modernize     Assess codebase maturity + plan         modernize
/r:util:test-dedup         Find duplicate/overlapping tests        test-dedup
/r:util:chg-gen            Generate changelog entries from diff    rel-chg-diff
/r:util:chg-dedup          Deduplicate changelog entries           rel-chg-dedup
```

**Relationship to lifecycle commands:**

Some utilities overlap with lifecycle command internals. For example,
`/r:ship` internally runs changelog generation and deduplication. The
utilities exist for cases where users want to run these operations
independently — outside of a release, during ad-hoc maintenance, or
as part of a custom workflow.

Utilities are implemented as skills (main context). They do not dispatch
subagents.

### /r:audit Workflow

`/r:audit` performs a comprehensive codebase audit by dispatching multiple
reviewer subagents in parallel, each with a different focus:

```
/r:audit
  |
  +-- Read governance index + verification.md + anti-patterns.md
  |
  +-- Dispatch parallel reviewer subagents:
  |     reviewer --sentinel (regression + anti-slop)
  |     reviewer --sentinel (security focus)
  |     reviewer --sentinel (performance focus)
  |     qa (lint + standards verification)
  |
  +-- Collect results
  |
  +-- Synthesize into structured findings report (AUDIT.md)
  |     Severity: critical / major / minor
  |     Category: regression / security / performance / standards
  |     Deduplicated across reviewer passes
  |
  +-- Present summary to user with actionable items
```

This differs from `/r:review` (which reviews a specific diff or spec)
in that `/r:audit` scans the **entire codebase** for latent issues,
not just recent changes.

### /r:ship Workflow

`/r:ship` is a composite workflow for release preparation:

```
/r:ship
  |
  +-- PREFLIGHT
  |   Verify all PLAN.md phases are complete
  |   Check for uncommitted changes
  |   Read governance/verification.md for project-specific release checks
  |
  +-- VERIFICATION (dispatches subagents)
  |   qa subagent: full verification suite
  |   reviewer --sentinel: final adversarial review of branch diff
  |
  +-- RELEASE PREP (main context)
  |   Generate/update changelog from commit history
  |   Scrub AI attribution (if governance requires it)
  |   Verify version strings are consistent
  |
  +-- PUBLISH (main context, with user confirmation)
  |   Create PR with structured description
  |   Monitor CI checks (if configured)
  |
  +-- Report final status to user
```

### /r:sync — RDF Internal Command

`/r:sync` is an RDF self-development command for maintaining the canonical
source. It pulls emergency edits made directly to `~/.claude/` back into
`rdf/canonical/`, stripping tool-specific frontmatter during import.
This command is included in the public surface because RDF contributors
need it, but it is not relevant to projects consuming RDF.

### Full Command Hierarchy

```
/r:{command}                13 lifecycle commands (workflow)
/r:util:{subject}-{verb}    10 utility commands (maintenance)
                            --
                            23 total (down from ~68)
```

Lifecycle commands are single words (`init`, `plan`, `build`, `ship`).
Utility commands are always `subject-verb` (`mem-compact`, `chg-gen`).
The `/r:` vs `/r:util:` prefix is the tier differentiator.

### What Disappeared

The ~45 commands removed from RDF 2.x fall into four categories:

1. **Absorbed by lifecycle commands**: All `audit-*` sub-agents
   (24 commands) become internal to `/r:audit`. Release sub-steps
   (`rel-prep`, `rel-merge`, `rel-squash`, `rel-notes`, `rel-scrub`)
   become internal to `/r:ship`. Memory management (`mem-save`,
   `mem-audit`) becomes internal to `/r:refresh`. Code quality
   (`code-validate`, `test-strategy`, `test-impact`) becomes internal
   to the dispatcher and QA agent.
2. **Promoted to utility tier**: 10 standalone tools that remain useful
   outside lifecycle workflows moved to `/r:util:` with normalized
   naming (see Utility Commands above).
3. **Unnecessary with generic agents**: All `sys-*`, `fe-*`, `sec-*`
   persona dispatchers (12 commands) are gone. Domain-prefixed persona
   commands served no purpose when agents are universal.
4. **Deprecated stubs removed**: `audit-dedup`, `audit-synthesis`
   (already deprecated in 2.x).

## Section 6: Profiles

Profiles shift from "which agents to bundle" to "which governance
templates and reference docs to provide." They inform `/r:init`, not
the agents themselves.

### Profile Structure

```
profiles/
  core/                          <-- always active, minimal
    governance-template.md       <-- base verification checks, commit protocol
    reference/
      framework.md               <-- RDF artifact taxonomy

  systems-engineering/           <-- bash/shell/Linux projects
    governance-template.md       <-- shell standards, portability, bash compat
    reference/
      os-compat.md               <-- target OS matrix
      test-infra.md              <-- BATS framework details

  frontend/
    governance-template.md       <-- accessibility, component testing, browser compat
    reference/
      browser-matrix.md
      design-system.md

  security/
    governance-template.md       <-- OWASP checks, threat modeling, audit requirements
    reference/
      threat-model-template.md

  python/                        <-- future
    governance-template.md       <-- typing, venv, pytest conventions

  full-stack/                    <-- future, composes others
    requires: [frontend, python]
    governance-template.md       <-- cross-layer integration patterns
```

### How Profiles Interact with /r:init

```
/r:init detects Python + React project
  |
  +-- Auto-suggests: "Detected Python backend + React frontend.
  |   Recommend profiles: core + python + frontend. Accept? [Y/n]"
  |
  +-- User confirms (or overrides)
  |
  +-- Profile governance templates become SEED DATA for governance generation
  |   They are not copied verbatim -- they are merged with what the codebase
  |   scan actually finds. If the template says "use pytest" but the project
  |   uses unittest, the scan wins.
  |
  +-- Profile reference docs are copied into .claude/governance/reference/
      Available for just-in-time loading by agents
```

### Key Changes from RDF 2.x

- Profiles no longer list agents (all agents are universal)
- Profiles no longer list commands (all commands are universal)
- Profiles are **seed templates** for governance generation, not
  deployment manifests
- Profile detection can be automatic (from codebase scan) rather than
  manual selection
- Multiple profiles compose naturally — full-stack is just
  frontend + backend merged
- Outside stakeholders never need to think about profiles —
  `/r:init` handles it

## Section 7: Operational Modes

Project governance is intrinsic — how the project is built. Operational
modes are extrinsic — what the operator is doing right now.

### The Two Axes

```
PROJECT GOVERNANCE (intrinsic)          OPERATIONAL MODE (extrinsic)
"How this project is built"             "What I'm doing right now"
-------------------------------         --------------------------
Coding conventions                      Development (default)
Platform constraints                    Security assessment
Test framework                          Performance audit
Build system                            Migration planning
Anti-patterns                           Compliance review
```

### Mode Structure

```
modes/
  development/          <-- default, implicit
    context.md          <-- TDD protocol, quality gates, commit standards

  security-assessment/
    context.md          <-- OWASP methodology, threat modeling checklist,
                           attack surface enumeration, finding severity schema

  performance-audit/
    context.md          <-- profiling methodology, benchmark protocol,
                           bottleneck classification

  migration/
    context.md          <-- compatibility matrix, rollback planning,
                           data migration verification
```

### Mode Activation

```
/r:init . --mode security-assessment   <-- set mode at init time
/r:mode security                       <-- switch mode in existing session
```

### How Modes Affect Agents

Modes change how agents behave, not what they know about the project.

- `/r:plan` in security mode: planner brainstorms attack vectors, not features
- `/r:build` in security mode: dispatcher applies security-specific quality gates
- `/r:review --sentinel` in security mode: reviewer uses security-focused checklist

Same agents, same project governance, different operational context.

Modes are session-scoped and do not modify governance files.

## Section 8: Context Management & Reload

### Always Loaded (~100-150 tokens)

```
.claude/governance/index.md
  - One-line descriptions of each governance file
  - One-line descriptions of each existing .md file
  - Current operational mode (if set)
  - Pointer to PLAN.md current phase (if exists)
```

### Session Initialization (/r:start)

Reads index.md, displays status anchor:

```
+-- Context Anchor ------------------------------------------------+
| Project: fastapi-app       Branch: feature/websockets            |
| HEAD:    a3f21bc           Dirty: 2 files                        |
| Plan:    3/7 phases        Mode: development                     |
| Governance: 6 files        Age: 2h since refresh                 |
+------------------------------------------------------------------+
```

No governance file contents loaded — just the anchor. Fast, cheap.

### Just-in-Time Loading by Agent

Each agent reads the governance index and loads only what it needs:

| Agent | Loads |
|-------|-------|
| Planner | architecture.md, constraints.md |
| Engineer | conventions.md, constraints.md, anti-patterns.md |
| QA | verification.md |
| Reviewer | anti-patterns.md, constraints.md |
| UAT | architecture.md |

### Compaction Resilience

The governance index is ~100 tokens — small enough to survive any
compaction event. After compaction, an agent re-reads the index and
re-loads what it needs. No special compaction hooks required.

Plan state (current phase, completed phases) is in PLAN.md on disk —
never depends on conversation history surviving compaction.

Status outputs from `/r:build` are written to disk (`work-output/`) —
the dispatcher reads them back after compaction to resume.

### Cross-Session Continuity

```
Session 1: /r:plan writes spec + PLAN.md to disk
Session 2: /r:start reads index -> sees PLAN.md -> shows phase progress
           /r:build picks up where session 1 left off
```

No MEMORY.md dependency for operational state. MEMORY.md remains optional
for human-curated project context that does not belong in governance
(team decisions, external references, stakeholder notes).

### Refresh Triggers

Run `/r:refresh` when:
- Governance age exceeds 24h (flagged in `/r:start`)
- Significant codebase changes (new framework, major refactor)
- User request

`/r:refresh` re-runs codebase scan, updates governance files, preserves
user modifications, flags drift between governance and reality.

STEP 2: INIT A PROJECT
  cd /path/to/any-project

  # Output:
  # +-- RDF Init ----------------------------------------------------+
  # | Ingested: CLAUDE.md (247 lines), AGENTS.md (89 lines)         |

  # |   .claude/governance/anti-patterns.md  (from CLAUDE.md)        |
  # |                                                                |

  # |   ! architecture.md: unclear if services/ is monolith          |

STEP 3: WORK
  /r:plan          # brainstorm, research, spec, plan
  /r:build         # execute phase 1 with TDD and quality gates

### What Makes This Work for Outsiders

- **23 commands** in two tiers (`/r:` lifecycle + `/r:util:` maintenance)
  — learn the 13 lifecycle commands, utilities are there when you need them
- **Zero domain taxonomy** — no `sys-`, `fe-`, `sec-` to decode
- **Auto-detection** — profiles and conventions inferred, not configured
- **Familiar files stay** — existing CLAUDE.md is not replaced or reformatted
- **Low-confidence flagging** — init tells you what it is not sure about
- **Progressive depth** — `/r:plan` and `/r:build` handle 90% of work;
  other commands exist when you need fine-grained control

### Profile Migration

RDF 2.x profiles defined agent lists and command lists. RDF 3.0 profiles
contain only governance templates and reference docs. The profile
directory structure changes:

```
RDF 2.x:                          RDF 3.0:
  profiles/sys-eng/                 profiles/systems-engineering/
    profile.json (agents, cmds)       governance-template.md
    governance.md                     reference/
    reference/                          os-compat.md
    templates/                          test-infra.md
```

`profile.json` files are no longer needed — profiles do not control
agent or command availability.

### Backward Compatibility

RDF 3.0 is a breaking change. The `/r:` namespace, universal agents,
and governance-first model are incompatible with RDF 2.x's domain-prefixed
agents and static profile manifests.

For rfxn projects already using RDF 2.x:
1. Run `/r:init` — existing CLAUDE.md files are ingested as highest-priority
   governance input. All existing conventions are preserved.
2. Domain-specific knowledge currently in agent prompts moves to governance
   files automatically (generated from existing CLAUDE.md content).
3. Existing PLAN.md and MEMORY.md files are read during init.

## Research & Industry Alignment

This architecture is informed by research across the AI agent framework
ecosystem as of March 2026.

### Anthropic's Context Engineering Guidance

Anthropic's engineering team recommends: "Do the simplest thing that
works." Key principles applied:

- **Just-in-time context** over pre-loading (governance index + on-demand files)
- **Goldilocks system prompts** — specific enough to guide, flexible enough
  for heuristics (agent prompts define protocol, not domain knowledge)
- **Sub-agent isolation** — specialized subagents handle focused tasks,
  return condensed summaries (engineer, qa, uat, reviewer)

### Claude Code Native Architecture

RDF 3.0 aligns with Claude Code's own primitives:

- **Subagents**: Markdown files with YAML frontmatter. Domain-agnostic by
  design — system prompt defines the role, CLAUDE.md provides project context.
- **Agent Skills**: Progressive disclosure — metadata always loaded,
  instructions loaded when triggered, resources loaded as needed.
- **Agent Teams**: Team lead + 3-5 teammates with shared task list.
  Production sweet spot: 5-6 tasks per teammate.
- **Layered context**: CLAUDE.md hierarchy (user -> project -> local)
  provides domain knowledge. Agents are generic; context makes them specific.

### Industry Frameworks

- **CrewAI**: Agents defined by role/goal/backstory (generic), organized
  into crews. Validates the role-based model but shows limitations for
  conditional branching.
- **LangGraph**: Agents as graph nodes with shared state. More flexible
  but more complex. Subgraphs for modular, reusable pipelines.
- **Cursor**: Glob-based rule activation. Global rules for universal
  conventions, project rules for specifics. Keep under 300 lines.
- **Layered Configuration Context**: Four-layer cascade
  (enterprise -> user -> project -> local) documented as a formal
  agentic pattern.
