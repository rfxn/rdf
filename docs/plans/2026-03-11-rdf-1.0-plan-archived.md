- Symlinks to output directory make all commands/agents/hooks functional
- Zero behavioral difference from current direct `~/.claude/` layout

### Phase 3: Core Profile + Systems Engineering Profile

PENDING

**Goal:** Define the profile system and create the first two profiles.
Separate framework-level conventions from domain-specific conventions.

**Steps:**

1. Define `profiles/registry.json` schema:
   ```json
   {
     "version": "1.0",
     "profiles": {
       "core": {
         "name": "Core Framework",
         "description": "Framework primitives — memory, state, project management",
         "requires": []
       },
       "systems-engineering": {
         "name": "Systems Engineering",
         "description": "Bash/shell projects — conventions, testing, portability",
         "requires": ["core"]
       }
     }
   }
   ```
2. Create `profiles/core/profile.json`:
   - `commands`: [mem-save, mem-compact, mem-audit, proj-status, proj-health,
     reload, onboard, status, scope, modernize]
   - `agents`: [planner]
   - `scripts`: [context-bar.sh, clone-conversation.sh, half-clone-conversation.sh,
     check-context.sh]
   - `hooks`: [PreCompact, SessionStart]
3. Create `profiles/core/governance.md`:
   - Extract from parent CLAUDE.md: commit protocol, changelog tracking,
     documentation sync, MEMORY.md standards, session safety, artifact taxonomy,
     development workflow summary, common anti-patterns (tool-agnostic ones only)
   - This becomes the new tool-agnostic parent governance doc
4. Create `profiles/systems-engineering/profile.json`:
   - `commands`: [em, se, qa, uat, audit-*, rel-*, code-*, test-*, proj-cross,
     proj-lib-sync, proj-cross-audit, ci-setup, lib-release, proj-scaffold,
     code-grep, code-validate, planner, frontend-qa, frontend-uat]
   - `agents`: [se, qa, uat, frontend-qa, frontend-uat]
   - `scripts`: [pre-commit-validate.sh, post-edit-lint.sh, subagent-stop.sh,
     overwatch-hook.sh]
   - `hooks`: [PreToolUse (bash validation), PostToolUse (lint), SubagentStop]
5. Create `profiles/systems-engineering/governance.md`:
   - Extract from parent CLAUDE.md: coding conventions (bash standards, shell
     standards, shellcheck directives, binary paths, error handling, fix quality),
     CLI constraint (frozen), portability & compatibility (OS matrix, bash 4.1
     floor, AWK compat), testing standards (BATS/batsman, test tiers, test
     quality gates), verification before commit, architecture patterns
6. Move domain reference docs:
   - `/root/admin/work/proj/reference/os-compat.md` → `profiles/systems-engineering/reference/`
   - `/root/admin/work/proj/reference/test-infra.md` → same
   - `/root/admin/work/proj/reference/cross-project.md` → same
   - `/root/admin/work/proj/reference/audit-pipeline.md` → same
7. Create governance templates:
   - `profiles/systems-engineering/templates/claude-shell.md.tmpl`
   - `profiles/systems-engineering/templates/claude-lib.md.tmpl`
   - Templates use `{{PROJECT_NAME}}`, `{{VERSION}}`, `{{DESCRIPTION}}` variables
8. Write `bin/rdf-profile`:
   - `rdf-profile list` — read registry.json, display profiles with descriptions
   - `rdf-profile install <name> [<name>...]` — resolve deps, write
     `~/.rdf/active-profiles.json`, trigger `rdf-generate` for active tool
   - `rdf-profile remove <name>` — deactivate, warn if dependents active
   - `rdf-profile status` — show active profiles with component counts
9. Update `rdf-generate` to be profile-aware:
   - Only include commands/agents/scripts from active profiles' component lists
   - Concatenate governance.md from each active profile (core first, then others)
   - Generate composite CLAUDE.md for tool-specific output
10. Verify: `rdf-profile install core systems-engineering && rdf-generate claude-code`
    produces identical output to Phase 2's full generation
11. Verify: parent CLAUDE.md content is fully covered between core + systems-engineering
    governance docs (nothing lost in the split)

**Acceptance criteria:**
- Profile registry defines core + systems-engineering
- `rdf-profile install/list/status/remove` all work
- `rdf-generate` respects active profiles
- Core governance + systems-engineering governance = complete parent CLAUDE.md content
- Domain reference docs live under profile, not framework-level

### Phase 4: Project Initializer (`rdf-init`)

PENDING

**Goal:** Repeatable project initialization that makes onboarding 46 inactive
projects feasible. Profile-aware, multi-tool.

**Steps:**

1. Write `bin/rdf-init`:
   - Arguments: `rdf-init <path> [--type shell|lib|frontend|minimal]
     [--tools claude-code,gemini-cli] [--version X.Y.Z] [--no-memory]`
   - `--batch` mode: `rdf-init --batch /root/admin/work/proj/inactive/*`
2. Project type detection (when `--type` not specified):
   - `files/` dir + Makefile → shell
   - `lib/` or single `.sh` + `tests/` → lib
   - `package.json` → frontend
   - Otherwise → minimal
3. CLAUDE.md generation:
   - Read active profiles
   - Select template matching project type from active profile's `templates/`
   - Substitute variables: `{{PROJECT_NAME}}` (basename), `{{VERSION}}` (from
     git tag, VERSION file, or `--version`), `{{DESCRIPTION}}` (from README
     first line, or placeholder)
   - Write to `<path>/CLAUDE.md`
4. AGENTS.md generation:
   - Generate cross-tool AGENTS.md from profile governance
   - Follow AGENTS.md spec: build commands, test commands, conventions
   - Under 150 lines
   - Write to `<path>/AGENTS.md`
5. Tool-specific files:
   - If `--tools` includes `gemini-cli`: generate `GEMINI.md`
6. Git exclude setup:
   - Read `<path>/.git/info/exclude` (create if missing)
   - Append entries only if not already present: CLAUDE.md, PLAN*.md,
     AUDIT.md, MEMORY.md, .claude/, work-output/, audit-output/
7. Create `<path>/work-output/` directory
8. Memory generation (unless `--no-memory`):
   - Dispatch `/onboard` skill/command to read project and generate MEMORY.md
   - Write to auto-memory location or `<path>/MEMORY.md`
9. Report: list all created files, sizes, any warnings
10. Batch mode:
    - Iterate over all paths, skip non-directories, skip if CLAUDE.md exists
      (unless `--force`)
    - Summary table: project, type detected, files created, warnings

**Acceptance criteria:**
- `rdf-init` creates correct CLAUDE.md from profile template
- AGENTS.md generated following cross-tool spec
- .git/info/exclude entries added non-destructively
- Batch mode processes multiple directories
- `--no-memory` skips agent dispatch
- Idempotent: running twice on same project produces no changes (no duplicates)

### Phase 5: Health Checker (`rdf-doctor`)

PENDING

**Goal:** Convention drift detection and health verification at scale.
Run across all projects to find what's stale, missing, or divergent.

**Steps:**

1. Write `bin/rdf-doctor`:
   - Arguments: `rdf-doctor [<path>] [--scope all|drift|memory|plan|artifacts|sync]`
   - `--all` flag: run across all projects in workspace
2. Artifact presence check:
   - Does CLAUDE.md exist?
   - Does AGENTS.md exist? (if cross-tool configured)
   - Does .git/info/exclude have required entries?
   - Does work-output/ exist?
   - Does MEMORY.md exist? (if project has active work — commits in last 30 days)
3. Convention drift check:
   - Diff project CLAUDE.md against current profile template (ignoring
     project-specific sections)
   - Flag conventions that are present in profile governance but missing
     from project CLAUDE.md
   - Flag project CLAUDE.md content that contradicts profile governance
4. Memory staleness check:
   - Compare MEMORY.md last-modified vs latest git commit timestamp
   - Check if version string in MEMORY.md matches actual VERSION/git tag
   - Check if test counts are current (compare against last test run if cached)
5. Plan coherence check:
   - Read PLAN.md phase status markers
   - Cross-reference against `git log` — phases marked PENDING that have
     corresponding commits should be flagged
   - Phases marked COMPLETED without commit hashes should be flagged
6. Tool sync check:
   - If CLAUDE.md and AGENTS.md both exist, verify they were generated from
     same profile version (check header comment with generation timestamp)
   - Flag AGENTS.md that's older than last `rdf-generate` run
7. Cross-project checks (with `--all`):
   - Shared library version consistency across consumers
   - Profile version consistency (all projects on same profile version?)
8. Output format:
   ```
   PROJECT          ARTIFACTS  DRIFT  MEMORY  PLAN  SYNC  OVERALL
   apf              OK         WARN   STALE   OK    OK    WARN
   bfd              OK         OK     OK      OK    OK    OK
   overwatch        MISSING    FAIL   MISSING N/A   N/A   FAIL
   inactive/les-0.2 MISSING    N/A    MISSING N/A   N/A   FAIL
   ```
9. Suggested fixes:
   - For each WARN/FAIL, output actionable one-liner
   - Example: `STALE MEMORY: run 'rdf-refresh bfd' or '/mem-save'`
   - Example: `MISSING ARTIFACTS: run 'rdf-init /path/to/les-0.2 --type shell'`

**Acceptance criteria:**
- Correctly detects all 5 check categories
- `--all` scans all projects in workspace
- Output is a clear table with per-project status
- Suggested fixes are actionable
- False positive rate <10% (verify against known-good projects)

### Phase 6: State Helper + Refresh Skill

PENDING

**Goal:** Deterministic state checking (no LLM tokens wasted on git commands)
and agent-driven state file refresh.

**Steps:**

1. Write `state/rdf-state.sh`:
   - Input: project path (default: current directory)
   - Output: JSON to stdout
   - Fields:
     ```json
     {
       "project": "bfd",
       "path": "/root/admin/work/proj/brute-force-detection",
       "version": "2.0.1",
       "branch": "2.0.1",
       "dirty": false,
       "uncommitted_files": 0,
       "last_commit_hash": "abc1234",
       "last_commit_age_hours": 2,
       "commits_since_tag": 42,
       "memory_exists": true,
       "memory_age_hours": 12,
       "plan_exists": true,
       "plan_phases": { "total": 9, "completed": 7, "active": 1, "pending": 1 },
       "audit_exists": false,
       "work_output_files": ["phase-7-result.md", "qa-phase-7-verdict.md"],
       "active_profiles": ["core", "systems-engineering"],
       "tools_configured": ["claude-code"]
     }
     ```
   - Implementation: pure bash + `git` + `jq` (or manual JSON construction)
   - No LLM involvement — deterministic, <1 second execution
   - ~150 lines of bash
2. Integrate with EM:
   - EM startup reads `rdf-state` output instead of running 5+ bash commands
   - Saves ~500-1000 tokens per EM session startup
   - Update `canonical/commands/em.md` to document rdf-state integration
3. Write `/refresh` as canonical command (`canonical/commands/refresh.md`):
   - Modes: `rdf-refresh [<path>] [--scope memory|plan|all]`
   - `memory` scope:
     - Read `rdf-state` output for current project state
     - Dispatch agent to update MEMORY.md sections: Status, Known Issues,
       Lessons Learned
     - Agent reads git log since last recorded commit, updates accordingly
   - `plan` scope:
     - Read PLAN.md phase list
     - Cross-reference against git log for commits mentioning phase numbers
     - Mark phases COMPLETED with commit hashes
     - Mark phases IN PROGRESS if partial commits exist
   - `all` scope: run both
   - Non-destructive: show diff before applying, require confirmation
4. Add `/refresh` to core profile component list
5. Verify: `rdf-state` returns accurate JSON for all 8 active projects
6. Verify: `/refresh --scope memory` correctly updates MEMORY.md
7. Verify: `/refresh --scope plan` correctly marks completed phases

**Acceptance criteria:**
- `rdf-state.sh` returns valid JSON in <1 second for any project
- `/refresh` updates MEMORY.md accurately from git state
- `/refresh` updates PLAN.md phase markers from git log
- No false positives (phases incorrectly marked complete)
- Works on projects with no MEMORY.md (creates initial version)
