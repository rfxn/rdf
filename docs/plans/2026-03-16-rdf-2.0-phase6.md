# RDF 2.0 — Phase 6 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development
> (if subagents available) or superpowers:executing-plans to implement this plan.
> Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Prove the multi-tool architecture by implementing Gemini CLI, Codex,
and AGENTS.md adapters — making RDF a genuinely tool-agnostic framework where
`rdf generate <target>` produces valid configuration for any supported tool.

**Architecture:** Each adapter follows the pattern established in Phase 2's
Claude Code adapter: a `adapter.sh` sourced by `lib/cmd/generate.sh` that reads
`canonical/` sources + adapter-local metadata, then writes tool-specific output
to `adapters/<target>/output/`. The `agents-md` adapter is unique — it produces
a single AGENTS.md file (not a directory tree) suitable for cross-tool discovery.

**Spec:** `docs/specs/2026-03-16-rdf-2.0-architecture-design.md` §3, §4, §8 Phase 6

**Tech Stack:** Bash 4.1+, JSON (jq), `gh` CLI, git

**Dependencies:** Phase 2 (adapter pattern + generate.sh), Phase 3 (profiles)

## Research Findings

### Gemini CLI Configuration Model

**Sources:**
- [Gemini CLI Configuration](https://geminicli.com/docs/reference/configuration/)
- [GEMINI.md Context Files](https://geminicli.com/docs/cli/gemini-md/)
- [Gemini CLI Subagents](https://geminicli.com/docs/core/subagents/)
- [Custom Commands](https://geminicli.com/docs/cli/custom-commands/)
- [Creating Agent Skills](https://geminicli.com/docs/cli/creating-skills/)
- [GitHub: google-gemini/gemini-cli](https://github.com/google-gemini/gemini-cli)

**Directory structure:**

```
~/.gemini/                              # User-level (global)
|-- settings.json                       # Global config (model, MCP, theme)
|-- GEMINI.md                           # Global context instructions
|-- agents/                             # User-level custom agents
|   +-- *.md                            # YAML frontmatter + system prompt
+-- commands/                           # User-level slash commands
    +-- *.toml                          # TOML: prompt + description + args

<project>/.gemini/                      # Project-level
|-- GEMINI.md                           # Project-specific context (equiv to CLAUDE.md)
|-- agents/                             # Project-level custom agents
|   +-- *.md                            # YAML frontmatter + system prompt
+-- commands/                           # Project-level slash commands
    +-- *.toml                          # TOML: prompt + description + args
```

**GEMINI.md format:**
- Plain markdown, no frontmatter required
- Hierarchical loading: `~/.gemini/GEMINI.md` -> each parent dir up to project
  root -> current dir -> subdirectories
- Supports `@file.md` import syntax for modularization (relative and absolute
  paths, `.md` files only)
- More specific files override general ones
- All found files concatenated and sent with every prompt

**Custom agent format (`.gemini/agents/*.md`):**
- YAML frontmatter (required) with `---` delimiters
- Fields: `name` (string, identifier), `description` (string, single-line —
  critical for trigger matching since body only loads after trigger)
- Optional: tool restrictions via settings.json
- Body = system prompt (markdown)
- Agents exposed as tools to the main agent (same name)
- Must enable custom subagents in settings.json

**Custom command format (`.gemini/commands/*.toml`):**
- TOML format (not markdown)
- Required key: `prompt` (string, single or multi-line)
- Optional: `description` (shown in `/help`), `{{args}}` template variable
- File path determines command name: `test.toml` -> `/test`,
  `git/commit.toml` -> `/git:commit` (subdirs create namespaces with `:`)
- User-scoped: `~/.gemini/commands/`, project-scoped: `.gemini/commands/`

**Key differences from Claude Code:**
- Agents use minimal frontmatter (`name`, `description` only — no `tools`,
  `model`, `disallowedTools` in frontmatter; tool config via settings.json)
- Commands use TOML, not markdown
- No plugin manifest concept
- Hooks configured via settings.json, not a separate hooks.json
- Context files use `GEMINI.md` (not `CLAUDE.md`)

### Codex CLI Configuration Model

**Sources:**
- [Custom instructions with AGENTS.md](https://developers.openai.com/codex/guides/agents-md/)
- [Codex CLI Configuration Reference](https://developers.openai.com/codex/config-reference/)
- [Codex CLI Config Basics](https://developers.openai.com/codex/config-basic/)
- [Codex CLI Sample Configuration](https://developers.openai.com/codex/config-sample/)
- [Codex CLI Advanced Configuration](https://developers.openai.com/codex/config-advanced/)

**Directory structure:**

```
~/.codex/                               # User-level (global)
|-- config.toml                         # Global config (model, sandbox, etc.)
|-- AGENTS.md                           # Global agent instructions
+-- AGENTS.override.md                  # Override (takes precedence)

<project>/.codex/                       # Project-level
+-- config.toml                         # Project config overrides

<project>/AGENTS.md                     # Project-level agent instructions
<project>/AGENTS.override.md            # Override (takes precedence)
```

**AGENTS.md format:**
- Plain markdown, no frontmatter
- Standard heading-based sections: `# Build & Test`, `# Architecture Overview`,
  `# Security`, `# Git Workflows`, `# Conventions & Patterns`
- Discovery hierarchy: `~/.codex/` -> project root -> walk down to CWD
- Fallback filenames checked in order: `AGENTS.override.md`, `AGENTS.md`,
  `TEAM_GUIDE.md`, `.agents.md`
- Size limit: 32 KiB default (`project_doc_max_bytes`)
- Each file becomes a user-role message prefixed with
  `# AGENTS.md instructions for <directory>`
- Nested: place AGENTS.md in subdirectories for package-specific instructions

**Configuration format:**
- Primary: `config.toml` (TOML)
- Dot notation for nested values
- CLI overrides parsed as TOML/JSON

**Key differences from Claude Code:**
- No separate agent definitions — single AGENTS.md file contains all context
- No custom command files (commands are part of AGENTS.md instructions)
- No plugin manifest, no hooks system
- Config is TOML-based (not JSON)
- Much simpler model: one governance document, not individual agent personas

### AGENTS.md Cross-Tool Standard

**Sources:**
- [AGENTS.md Spec (GitHub)](https://github.com/agentsmd/agents.md)
- [AGENTS.md Official Site](https://agents.md/)
- [Factory AGENTS.md Docs](https://docs.factory.ai/cli/configuration/agents-md)

**Status:** Governed by Linux Foundation's Agentic AI Foundation (AAIF) since
December 2025. Members: OpenAI, Anthropic, Google, AWS, Bloomberg, Cloudflare.
Over 60,000 open-source repos on GitHub include AGENTS.md.

**Format:**
- Plain markdown with semantic headings
- No required schema — headings provide hints, agent parses text
- Guideline: aim for under 500 lines per file
- Place throughout project hierarchy; closest file takes precedence

**Common sections:**
```markdown
# Project Context
# Build & Test Commands
# Architecture Overview
# Code Style & Conventions
# Security
# Git Workflows
# Constraints & Boundaries
```

**Cross-tool compatibility:** Codex, Gemini CLI, Cursor, Aider, Zed, Jules,
Factory Droids, Phoenix, Kilo Code, and growing.

**RDF adapter strategy:** Generate a single AGENTS.md from RDF governance docs
+ profile rules + canonical agent descriptions. This gives any AGENTS.md-aware
tool immediate project context without requiring tool-specific adapters.

## Chunk 2: Gemini CLI Adapter

### Task 3: Write adapters/gemini-cli/adapter.sh

**Files:**
- Create: `adapters/gemini-cli/adapter.sh`

Core generation logic for Gemini CLI. Produces:
1. `.gemini/agents/*.md` — YAML frontmatter + canonical body
2. `.gemini/commands/*.toml` — TOML format with canonical body as prompt
3. `.gemini/GEMINI.md` — project governance from active profiles
4. `scripts/` — direct copy (shell scripts are tool-agnostic)

Output layout mirrors Gemini CLI's expected discovery paths:

```
adapters/gemini-cli/output/
|-- .gemini/
|   |-- GEMINI.md                   # Governance (from profiles)
|   |-- agents/
|   |   +-- *.md                    # YAML frontmatter + system prompt
|   +-- commands/
|       +-- *.toml                  # TOML command definitions
+-- scripts/                        # Hook scripts (tool-agnostic)
    +-- *.sh
```

- [ ] **Step 3.1: Write adapter.sh**

```bash
#!/usr/bin/env bash
# adapters/gemini-cli/adapter.sh — Gemini CLI adapter
# (C) 2026 R-fx Networks <proj@rfxn.com>
# GNU GPL v2
# Sourced by lib/cmd/generate.sh — do not execute directly

# Requires: RDF_HOME, RDF_CANONICAL, RDF_ADAPTERS, jq

_GEM_ADAPTER_DIR="${RDF_ADAPTERS}/gemini-cli"
_GEM_OUTPUT_DIR="${_GEM_ADAPTER_DIR}/output"
_GEM_AGENT_META="${_GEM_ADAPTER_DIR}/agent-meta.json"
_GEM_COMMAND_META="${_GEM_ADAPTER_DIR}/command-meta.json"

# Generate YAML frontmatter for a Gemini CLI agent
# Args: $1 = canonical agent basename (no extension)
# Output: YAML frontmatter to stdout, or return 1 if agent not in metadata
_gem_agent_frontmatter() {
    local agent="$1"
    local name desc

    # Check if agent exists in metadata
    if ! jq -e --arg a "$agent" '.[$a]' "$_GEM_AGENT_META" >/dev/null 2>&1; then
        rdf_warn "no Gemini metadata for agent: $agent — copying without frontmatter"
        return 1
    fi

    name="$(jq -r --arg a "$agent" '.[$a].name' "$_GEM_AGENT_META")"
    desc="$(jq -r --arg a "$agent" '.[$a].description' "$_GEM_AGENT_META")"

    echo "---"
    echo "name: ${name}"
    echo "description: \"${desc}\""
    echo "---"
}

# Generate all Gemini CLI agent files
# Reads canonical/agents/*.md + agent-meta.json -> output/.gemini/agents/*.md
gem_generate_agents() {
    local src_dir="${RDF_CANONICAL}/agents"
    local dst_dir="${_GEM_OUTPUT_DIR}/.gemini/agents"
    local count=0

    command mkdir -p "$dst_dir"

    for src_file in "${src_dir}"/*.md; do
        [[ -f "$src_file" ]] || continue
        local basename_f
        basename_f="$(basename "$src_file" .md)"
        local dst_file="${dst_dir}/${basename_f}.md"

        # Generate frontmatter + canonical body
        if _gem_agent_frontmatter "$basename_f" > "${dst_file}.tmp" 2>/dev/null; then
            echo "" >> "${dst_file}.tmp"
            cat "$src_file" >> "${dst_file}.tmp"
            command mv "${dst_file}.tmp" "$dst_file"
        else
            # No metadata — copy canonical body with minimal frontmatter
            {
                echo "---"
                echo "name: ${basename_f}"
                echo "description: \"RDF agent: ${basename_f}\""
                echo "---"
                echo ""
                cat "$src_file"
            } > "$dst_file"
        fi
        count=$((count + 1))
    done
    rdf_log "gemini-cli: generated ${count} agent files"
}

# Generate a single TOML command file from canonical markdown
# Args: $1 = canonical command markdown path, $2 = destination .toml path
_gem_write_command_toml() {
    local src_file="$1"
    local dst_file="$2"
    local basename_f
    basename_f="$(basename "$src_file" .md)"

    # Get description from metadata, or extract first sentence
    local desc
    desc="$(jq -r --arg c "$basename_f" '.[$c] // empty' "$_GEM_COMMAND_META" 2>/dev/null)"

    if [[ -z "$desc" ]]; then
        # Extract first non-empty line as description, truncate to 80 chars
        desc="$(sed -n '/^[^#[:space:]]/{ s/[[:space:]]*$//; p; q; }' "$src_file")"
        if [[ ${#desc} -gt 80 ]]; then
            desc="${desc:0:77}..."
        fi
        # Fallback if still empty
        [[ -z "$desc" ]] && desc="RDF command: ${basename_f}"
    fi

    # Read full canonical body as the prompt
    local body
    body="$(< "$src_file")"

    # Write TOML
    {
        echo "# Generated by rdf generate gemini-cli — do not edit"
        echo "# Source: canonical/commands/${basename_f}.md"
        echo ""
        echo "description = \"${desc}\""
        echo ""
        # Use TOML multi-line literal string for prompt
        echo 'prompt = """'
        echo "${body}"
        echo '"""'
    } > "$dst_file"
}

# Generate all Gemini CLI command files
# Reads canonical/commands/*.md -> output/.gemini/commands/*.toml
gem_generate_commands() {
    local src_dir="${RDF_CANONICAL}/commands"
    local dst_dir="${_GEM_OUTPUT_DIR}/.gemini/commands"
    local count=0

    command mkdir -p "$dst_dir"

    for src_file in "${src_dir}"/*.md; do
        [[ -f "$src_file" ]] || continue
        local basename_f
        basename_f="$(basename "$src_file" .md)"
        local dst_file="${dst_dir}/${basename_f}.toml"

        _gem_write_command_toml "$src_file" "$dst_file"
        count=$((count + 1))
    done
    rdf_log "gemini-cli: generated ${count} command TOML files"
}

# Generate GEMINI.md from profile governance docs
# Produces a single GEMINI.md with @import references for modular loading,
# or a concatenated single file if imports are not practical
gem_generate_context() {
    local dst_file="${_GEM_OUTPUT_DIR}/.gemini/GEMINI.md"
    local profiles_dir="${RDF_HOME}/profiles"

    command mkdir -p "$(dirname "$dst_file")"

    {
        echo "# RDF Project Governance — Gemini CLI Context"
        echo ""
        echo "Generated by \`rdf generate gemini-cli\` — do not edit directly."
        echo "Source: rdf/profiles/*/governance.md"
        echo ""
    } > "$dst_file"

    # Concatenate active profile governance docs
    # Core always included; others based on active profiles
    local profile_list="${RDF_HOME}/state/active-profiles"
    local profiles=("core")

    if [[ -f "$profile_list" ]]; then
        while IFS= read -r p; do
            [[ -z "$p" ]] && continue
            [[ "$p" == "core" ]] && continue
            profiles+=("$p")
        done < "$profile_list"
    fi

    for profile in "${profiles[@]}"; do
        local gov_file="${profiles_dir}/${profile}/governance.md"
        if [[ -f "$gov_file" ]]; then
            {
                echo ""
                echo "---"
                echo ""
                cat "$gov_file"
            } >> "$dst_file"
        else
            rdf_warn "gemini-cli: governance.md not found for profile: $profile"
        fi
    done

    rdf_log "gemini-cli: generated GEMINI.md (profiles: ${profiles[*]})"
}

# Generate script files (direct copy — scripts are tool-agnostic)
gem_generate_scripts() {
    local src_dir="${RDF_CANONICAL}/scripts"
    local dst_dir="${_GEM_OUTPUT_DIR}/scripts"
    local count=0

    command mkdir -p "$dst_dir"

    for src_file in "${src_dir}"/*.sh; do
        [[ -f "$src_file" ]] || continue
        local basename_f
        basename_f="$(basename "$src_file")"
        command cp "$src_file" "${dst_dir}/${basename_f}"
        chmod +x "${dst_dir}/${basename_f}"
        count=$((count + 1))
    done
    rdf_log "gemini-cli: generated ${count} script files"
}

# Full Gemini CLI generation pipeline
gem_generate_all() {
    rdf_log "generating Gemini CLI adapter output..."
    rdf_require_dir "$RDF_CANONICAL" "canonical directory"
    rdf_require_file "$_GEM_AGENT_META" "gemini-cli agent-meta.json"
    rdf_require_bin jq

    # Clean output directory
    command rm -rf "$_GEM_OUTPUT_DIR"
    command mkdir -p "$_GEM_OUTPUT_DIR"

    gem_generate_agents
    gem_generate_commands
    gem_generate_context
    gem_generate_scripts

    local agent_count command_count script_count
    agent_count="$(find "${_GEM_OUTPUT_DIR}/.gemini/agents" -name '*.md' 2>/dev/null | wc -l)"
    command_count="$(find "${_GEM_OUTPUT_DIR}/.gemini/commands" -name '*.toml' 2>/dev/null | wc -l)"
    script_count="$(find "${_GEM_OUTPUT_DIR}/scripts" -name '*.sh' 2>/dev/null | wc -l)"

    rdf_log "Gemini CLI generation complete: ${agent_count} agents, ${command_count} commands, ${script_count} scripts"
}
```

## Chunk 3: Codex Adapter

### Task 4: Write adapters/codex/adapter.sh

**Files:**
- Create: `adapters/codex/adapter.sh`

Codex uses a fundamentally different model from Claude Code and Gemini CLI:
there are no individual agent files or command files. Instead, Codex reads a
single AGENTS.md file (or hierarchy of them) for all project context. The
Codex adapter generates:

1. `AGENTS.md` — comprehensive project context document at project root level
2. `.codex/config.toml` — minimal Codex configuration (model preferences)
3. `scripts/` — direct copy (tool-agnostic)

The AGENTS.md produced by the Codex adapter is **more detailed** than the one
from the `agents-md` adapter because it targets Codex specifically and can
include Codex-specific sections (e.g., sandbox preferences, model hints).

Output layout:

```
adapters/codex/output/
|-- AGENTS.md                       # Full project context for Codex
|-- .codex/
|   +-- config.toml                 # Codex configuration
+-- scripts/                        # Hook scripts (tool-agnostic)
    +-- *.sh
```

- [ ] **Step 4.1: Write adapter.sh**

```bash

# adapters/codex/adapter.sh — Codex adapter
# (C) 2026 R-fx Networks <proj@rfxn.com>
# GNU GPL v2
# Sourced by lib/cmd/generate.sh — do not execute directly

# Requires: RDF_HOME, RDF_CANONICAL, RDF_ADAPTERS

_CDX_ADAPTER_DIR="${RDF_ADAPTERS}/codex"
_CDX_OUTPUT_DIR="${_CDX_ADAPTER_DIR}/output"

# Generate AGENTS.md for Codex
# Combines: governance docs + agent roster + command catalog + conventions
cdx_generate_agents_md() {

    local profiles_dir="${RDF_HOME}/profiles"

    command mkdir -p "$(dirname "$dst_file")"

    {
        echo "# AGENTS.md — rfxn Project Instructions for Codex"
        echo ""
        echo "Generated by \`rdf generate codex\` — do not edit directly."
        echo "Source: rdf/canonical/ + rdf/profiles/"

        echo "## Project Governance"
        echo ""
    } >> "$dst_file"

        local gov_file="${profiles_dir}/${profile}/governance.md"

            cat "$gov_file" >> "$dst_file"

        fi

    echo "Total commands available: ${cmd_count}" >> "$dst_file"
    echo "" >> "$dst_file"
```

```bash
#!/usr/bin/env bash
# adapters/agents-md/adapter.sh — AGENTS.md cross-tool adapter
# (C) 2026 R-fx Networks <proj@rfxn.com>
# GNU GPL v2
# Sourced by lib/cmd/generate.sh — do not execute directly

# Requires: RDF_HOME, RDF_CANONICAL, RDF_ADAPTERS, jq

_AMD_ADAPTER_DIR="${RDF_ADAPTERS}/agents-md"
_AMD_OUTPUT_DIR="${_AMD_ADAPTER_DIR}/output"
_AMD_SECTIONS="${_AMD_ADAPTER_DIR}/sections.json"

# Extract a section summary from a governance doc
# This produces a condensed version suitable for AGENTS.md
# Args: $1 = governance file path, $2 = section keyword to extract
_amd_extract_governance_section() {
    local gov_file="$1"
    local keyword="$2"

    if [[ ! -f "$gov_file" ]]; then
        echo "(governance file not found)"
        return 1
    fi

    # Find the section by keyword in headings, extract content until next heading
    local in_section=0
    local lines_out=0
    local max_lines=15

    while IFS= read -r line; do
        # Check for heading containing keyword (case-insensitive)
        local lower_line
        lower_line="$(echo "$line" | tr '[:upper:]' '[:lower:]')"
        local lower_key
        lower_key="$(echo "$keyword" | tr '[:upper:]' '[:lower:]')"

        if [[ "$lower_line" =~ ^##.*${lower_key} ]]; then
            in_section=1
            continue
        fi

        # Stop at next heading of same or higher level
        if [[ $in_section -eq 1 ]] && [[ "$line" =~ ^## ]]; then
            break
        fi

        if [[ $in_section -eq 1 ]] && [[ -n "$line" ]]; then
            echo "$line"
            lines_out=$((lines_out + 1))
            if [[ $lines_out -ge $max_lines ]]; then
                break
            fi
        fi
    done < "$gov_file"
}

# Generate agent roster as bullet list
_amd_agent_roster() {
    for agent_file in "${RDF_CANONICAL}/agents"/*.md; do
        [[ -f "$agent_file" ]] || continue
        local agent_name
        agent_name="$(basename "$agent_file" .md)"
        local agent_desc
        agent_desc="$(sed -n '/^[^#[:space:]]/{ p; q; }' "$agent_file")"
        [[ -z "$agent_desc" ]] && agent_desc="Specialized agent"
        # Truncate long descriptions
        if [[ ${#agent_desc} -gt 100 ]]; then
            agent_desc="${agent_desc:0:97}..."
        fi
        echo "- **${agent_name}**: ${agent_desc}"
    done
}

# Full AGENTS.md generation
amd_generate_all() {
    rdf_log "generating AGENTS.md (cross-tool)..."
    rdf_require_dir "$RDF_CANONICAL" "canonical directory"
    rdf_require_file "$_AMD_SECTIONS" "sections.json"
    rdf_require_bin jq

    # Clean output directory
    command rm -rf "$_AMD_OUTPUT_DIR"
    command mkdir -p "$_AMD_OUTPUT_DIR"

    local dst_file="${_AMD_OUTPUT_DIR}/AGENTS.md"
    local profiles_dir="${RDF_HOME}/profiles"
    local max_lines
    max_lines="$(jq -r '.max_lines' "$_AMD_SECTIONS")"

    {
        echo "# AGENTS.md — rfxn Development Framework"
        echo ""
        echo "Cross-tool project instructions. Generated by \`rdf generate agents-md\`."
        echo ""
    } > "$dst_file"

    # Process each section from sections.json
    local section_count
    section_count="$(jq '.sections | length' "$_AMD_SECTIONS")"
    local i=0

    while [[ $i -lt $section_count ]]; do
        local heading source
        heading="$(jq -r ".sections[$i].heading" "$_AMD_SECTIONS")"
        source="$(jq -r ".sections[$i].source" "$_AMD_SECTIONS")"

        echo "## ${heading}" >> "$dst_file"
        echo "" >> "$dst_file"

        case "$source" in
            static)
                local content
                content="$(jq -r ".sections[$i].content" "$_AMD_SECTIONS")"
                echo "$content" >> "$dst_file"
                ;;
            governance)
                local profile extract gov_file
                profile="$(jq -r ".sections[$i].profile" "$_AMD_SECTIONS")"
                extract="$(jq -r ".sections[$i].extract" "$_AMD_SECTIONS")"
                gov_file="${profiles_dir}/${profile}/governance.md"
                _amd_extract_governance_section "$gov_file" "$extract" >> "$dst_file"
                ;;
            agents)
                _amd_agent_roster >> "$dst_file"
                ;;
            *)
                rdf_warn "agents-md: unknown section source: $source"
                ;;
        esac

        echo "" >> "$dst_file"
        i=$((i + 1))
    done

    # Verify line count
    local line_count
    line_count="$(wc -l < "$dst_file")"
    if [[ "$line_count" -gt "$max_lines" ]]; then
        rdf_warn "agents-md: AGENTS.md is ${line_count} lines (over ${max_lines} limit)"
    fi

    rdf_log "AGENTS.md generation complete: ${line_count} lines"
}
```

```bash
_generate_usage() {
    cat <<'USAGE'
Usage: rdf generate <target>

Build tool-specific output from canonical sources.

Targets:
  claude-code    Generate Claude Code adapter output
  gemini-cli     Generate Gemini CLI adapter output
  codex          Generate Codex adapter output (AGENTS.md + config)
  agents-md      Generate cross-tool AGENTS.md

# Source and run a single adapter
# Args: $1 = adapter name, $2 = adapter script relative to RDF_ADAPTERS,
#        $3 = generation function name
_generate_adapter() {
    local name="$1"
    local script="${RDF_ADAPTERS}/$2"
    local func="$3"

    if [[ ! -f "$script" ]]; then
        rdf_die "adapter not found: ${script}"
    fi

    # shellcheck disable=SC1090
    source "$script"
    "$func"
}

cmd_generate() {
    case "${1:-}" in
        claude-code)
            _generate_adapter "claude-code" "claude-code/adapter.sh" "cc_generate_all"
            ;;
        gemini-cli)
            _generate_adapter "gemini-cli" "gemini-cli/adapter.sh" "gem_generate_all"
            ;;
        codex)
            _generate_adapter "codex" "codex/adapter.sh" "cdx_generate_all"
            ;;
        agents-md)
            _generate_adapter "agents-md" "agents-md/adapter.sh" "amd_generate_all"
            ;;
        all)
            rdf_log "generating all adapters..."
            local failed=0

            # Claude Code
            if [[ -f "${RDF_ADAPTERS}/claude-code/adapter.sh" ]]; then
                _generate_adapter "claude-code" "claude-code/adapter.sh" "cc_generate_all" || failed=$((failed + 1))
            fi

            # Gemini CLI
            if [[ -f "${RDF_ADAPTERS}/gemini-cli/adapter.sh" ]]; then
                _generate_adapter "gemini-cli" "gemini-cli/adapter.sh" "gem_generate_all" || failed=$((failed + 1))
            fi

            # Codex
            if [[ -f "${RDF_ADAPTERS}/codex/adapter.sh" ]]; then
                _generate_adapter "codex" "codex/adapter.sh" "cdx_generate_all" || failed=$((failed + 1))
            fi

            # AGENTS.md
            if [[ -f "${RDF_ADAPTERS}/agents-md/adapter.sh" ]]; then
                _generate_adapter "agents-md" "agents-md/adapter.sh" "amd_generate_all" || failed=$((failed + 1))
            fi

            if [[ $failed -gt 0 ]]; then
                rdf_warn "${failed} adapter(s) failed"
            else
                rdf_log "all adapters generated successfully"
            fi
            ;;
        help|--help|-h)
            _generate_usage
            ;;
        "")
            rdf_die "missing target — run 'rdf generate help' for usage"
            ;;
        *)
            rdf_die "unknown target: $1 — run 'rdf generate help' for usage"
            ;;
    esac
}
```

## Design Decisions

### D1: Gemini CLI commands are TOML, not markdown

Gemini CLI requires `.toml` format for slash commands. The adapter converts
canonical markdown command bodies into TOML `prompt = """..."""` multi-line
strings. The `description` field comes from `command-meta.json` or first-
sentence extraction.

### D2: Gemini CLI agent descriptions are trigger-critical

Unlike Claude Code where descriptions are informational, Gemini CLI uses the
description as the sole trigger signal. The body (system prompt) only loads
after the agent is activated. Therefore, Gemini agent descriptions must be
comprehensive and include "when to use" guidance.

### D3: Codex adapter produces its own AGENTS.md (not reusing agents-md)

The Codex adapter generates a Codex-specific AGENTS.md that can include
Codex-aware sections (sandbox preferences, model hints, approval mode guidance).
The `agents-md` adapter produces a tool-neutral version. They share the same
canonical sources but differ in output specificity and size constraints.

### D4: AGENTS.md adapter has a 150-line hard cap

Per spec acceptance criteria. This forces aggressive summarization — the
adapter extracts key governance rules and agent one-liners rather than
including full governance docs. The section extraction function caps each
governance section at 15 lines.

### D5: Profile-aware generation via active-profiles state file

All three new adapters check `state/active-profiles` to determine which
governance docs to include. This file is managed by `rdf profile install/remove`
(Phase 3). If the file is missing, only `core` profile is used.

### D6: Script copying is uniform across all adapters

Hook scripts are tool-agnostic bash. All adapters that include scripts do a
direct copy from `canonical/scripts/`. The AGENTS.md adapter does not include
scripts (it is a single-file format).
