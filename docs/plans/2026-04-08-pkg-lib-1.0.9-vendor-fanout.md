## Phase 1 — Release pkg_lib v1.0.9

**Location:** `/root/admin/work/proj/pkg_lib`
**Branch:** `master`
**Deliverable:** one commit on `master`, one annotated tag `v1.0.9`, pushed to origin

### Steps

1. **Test gate (required before commit):**
   ```bash
   cd /root/admin/work/proj/pkg_lib
   make -C tests test 2>&1 | tee /tmp/test-pkg_lib-P1-debian12.log | tail -30
   grep "not ok" /tmp/test-pkg_lib-P1-debian12.log || echo "all tests passing"
   ```
   Gate: no `not ok` lines. If failures exist, abort and fix before re-running.

2. **Anvil gate (Rocky 9 cross-check for release):**
   ```bash
   DOCKER_HOST=tcp://192.168.2.189:2376 DOCKER_TLS_VERIFY=1 DOCKER_CERT_PATH=~/.docker/tls \
     make -C tests test-rocky9 2>&1 | tee /tmp/test-pkg_lib-P1-rocky9.log | tail -30
   grep "not ok" /tmp/test-pkg_lib-P1-rocky9.log || echo "all tests passing"
   ```
   Gate: no `not ok` lines.

3. **Lint sweep (CLAUDE.md §Verification Before Commit):**
   ```bash
   bash -n files/pkg_lib.sh
   shellcheck files/pkg_lib.sh
   ```
   Gate: both clean.

4. **Explicit staging — no `git add -A`:**
   ```bash
   git add files/pkg_lib.sh tests/09-config.bats CHANGELOG CHANGELOG.RELEASE
   git status --short
   ```
   Gate: staged list is exactly these four files. Nothing else.

5. **Commit (HEREDOC message format per CLAUDE.md):**
   ```bash
   git commit -m "$(cat <<'EOF'
   v1.0.9 | pkg_config_merge: anchor variable detection to shell assignment regex

   [Fix] pkg_config_merge: anchor variable detection to ^[[:space:]]*[A-Za-z_][A-Za-z0-9_]*=
   [New] Regression test: tests/09-config.bats bash conditional passthrough + bash -n parse check
   [Change] Function header documents merge remains unsafe for conditional assignments (branches collapse)
   EOF
   )"
   ```

6. **Tag (annotated):**
   ```bash
   git tag -a v1.0.9 -m "pkg_lib v1.0.9"
   ```

7. **Push (branch + tag):**
   ```bash
   git push origin master
   git push origin v1.0.9
   ```

### Verification

- `git log --oneline -1` shows the release commit
- `git tag -l v1.0.9` shows the tag
- `git show v1.0.9 --stat` lists exactly 4 files modified
- `grep -m1 '^pkg_lib_version=\|^PKG_LIB_VERSION=' files/pkg_lib.sh` returns `1.0.9`

### Rollback

If anything goes wrong before push:
- Commit not yet pushed: `git reset --soft HEAD^` to restore working tree, retry
- Tag created but not pushed: `git tag -d v1.0.9`, retry

If push succeeded and an issue is discovered later, file a pkg_lib v1.0.10 follow-up. **Never** rewrite pushed tags on master.

## Phase 2 — BFD vendor sync (PR #7)

**Location:** `/root/admin/work/proj/brute-force-detection`
**Branch:** `2.0.2`
**Deliverable:** one commit on `2.0.2`, pushed to PR #7


   Gate: empty output (files identical).

4. **Lint sweep:**
   ```bash
   bash -n files/internals/pkg_lib.sh
   shellcheck files/internals/pkg_lib.sh
   ```
   Gate: both clean.

5. **Anvil test gate — Debian 12 + Rocky 9 (CLAUDE.md minimum):**
   ```bash
   DOCKER_HOST=tcp://192.168.2.189:2376 DOCKER_TLS_VERIFY=1 DOCKER_CERT_PATH=~/.docker/tls \
     make -C tests test 2>&1 | tee /tmp/test-bfd-P2-debian12.log | tail -30
   grep "not ok" /tmp/test-bfd-P2-debian12.log || echo "all tests passing"

   ```
   Gate: no `not ok` lines on either.

6. **Explicit staging:**
   ```bash
   git add files/internals/pkg_lib.sh
   git status --short
   ```
   Gate: staged list is exactly `M files/internals/pkg_lib.sh`. Nothing else.

7. **Commit:**
   ```bash
   git commit -m "$(cat <<'EOF'
   2.0.2 | Vendored pkg_lib 1.0.8 → 1.0.9: pkg_config_merge multi-var fix

   [Fix] Sync files/internals/pkg_lib.sh to pkg_lib v1.0.9

   ```

8. **Push to PR #7:**
   ```bash
   git push origin 2.0.2
   ```

### Verification

- `git log --oneline -1` shows the vendor sync commit on `2.0.2`
- `git show HEAD --stat` shows exactly one file modified: `files/internals/pkg_lib.sh`
- `gh -R rfxn/brute-force-detection pr checks 7` (wait ~1-2 min after push) shows CI still green
- PR #7 now includes the vendor sync commit in its diff

### Rollback

If tests fail after the vendor swap:
1. `git checkout HEAD -- files/internals/pkg_lib.sh` to restore
2. Investigate — this likely means BFD has a latent incompatibility with 1.0.9
3. File as a pkg_lib follow-up or a BFD-side fix before retrying

If the push lands but CI goes red on PR #7:
1. `git revert HEAD --no-edit` and push the revert
2. Do not force-push
3. Investigate the regression before retrying

## Notes for the executor

- **One commit per phase.** Do not batch pkg_lib + BFD + LMD into a single squash. Each project has its own release cadence and commit format.
- **Respect the commit message formats.** pkg_lib uses `v1.0.9 | …`, BFD uses `2.0.2 | …`, LMD uses `[Fix] …`. This is enforced by CLAUDE.md §Commit Protocol.
- **Never force-push** any of the branches. If anything goes wrong after a push, `git revert` and investigate.
- **Sequential execution recommended** — simplifies rollback if a consumer CI surfaces an unexpected regression. Parallel worktree dispatch is possible but offers no material speedup for this plan.
- The executor does not need subagent dispatch. This is a hands-on plan that one agent or operator can run top-to-bottom in a single session.
