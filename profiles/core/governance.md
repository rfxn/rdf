# Core Governance

> RDF core profile — tool-agnostic conventions that apply to all rfxn projects
> regardless of domain. Every other profile inherits these rules.

---

## Organization & Authorship

- **Org:** R-fx Networks <proj@rfxn.com> | **Author:** Ryan MacDonald <ryan@rfxn.com> | **License:** GNU GPL v2
- **Copyright:** New files use the current year only. Pre-existing range-format files use `(C) 2002-YYYY`.
- Update headers at version bumps or start of calendar year. All version/copyright headers must stay in sync.
- When changing CLI options, config variables, exit codes, or key paths: update man page, README, usage functions, config comments, and importconf in the same commit.

## Commit Protocol

1. **One commit per logical unit** — never batch phases
2. **Message format:** APF/BFD: `VERSION | Description` | LMD: `[Type] description` | workforce/RDF: free-form descriptive (no version prefix)
3. **Tag every body line:** `[New]` `[Change]` `[Fix]`
4. No `Co-Authored-By` lines; remove all Claude/Anthropic attribution before merge
5. Stage files explicitly by name — never `git add -A` or `git add .`
6. Never commit working files: `CLAUDE.md`, `PLAN*.md`, `AUDIT.md`, `MEMORY.md`, `.claude/` — exclude via `.git/info/exclude`, not `.gitignore`
7. Amend safety: verify HEAD with `git log --oneline -3` before `--amend`; if other commits exist above your target, create a fixup commit instead
8. Both `CHANGELOG` and `CHANGELOG.RELEASE` must be updated on every code-changing commit (APF, BFD, LMD only — workforce/RDF has no changelog). Phases that only touch excluded working files (e.g., AUDIT.md) produce no commit.
9. All agent, command, and script development happens in `rdf/canonical/`. Run `rdf generate claude-code` to deploy changes to `/root/.claude/`. Never edit `/root/.claude/` directly except for emergency fixes (follow up with `rdf sync` to pull changes back to canonical).

## MEMORY.md

Update after releases, audits, and completed phases. **200-line hard cap** (WARN at 180, OVER at 200 — lines beyond 200 are truncated from context). Use `/mem-compact` to archive stale entries.

When recording version strings or test counts, grep from source — never forward-copy from prior entries. Test counts and HEAD hashes are especially volatile:
```bash
grep '^VERSION=' files/<project>
grep -rc '@test' tests/*.bats | awk -F: '{s+=$2}END{print s}'
```
Run `/mem-save` at session end, including planning-only sessions that modified PLAN*.md. Skip for zero-change sessions (reload, model switch only).

## GitHub Conventions

### Workflow States

All Development Projects use the 5-state workflow on the Status field:

| State | Description | Transition From | Transition To |
|-------|-------------|----------------|---------------|
| Backlog | Not yet scheduled | — | Ready |
| Ready | Dependencies met, ready to pull | Backlog | In Progress |
| In Progress | Actively being worked | Ready | In Review |
| In Review | QA/UAT/peer review gate | In Progress | Done, In Progress (rework) |
| Done | Complete and verified | In Review | — |

When working phase issues, agents MUST transition status:
- `/sys-eng` picks up phase → set **In Progress**, post task-completion comments
- `/sys-eng` commits + reports → set **In Review**
- `/sys-qa` approves → set **Done**, close issue
- `/sys-qa` requests changes → set **In Progress** (rework)

### Issue Conventions (v2 — Phase-Level Tracking)

- Initiative issues (`type:initiative`) for roadmap planning
- Release issues (`type:release`) for versioned release tracking
- Phase issues (`type:phase`) contain detailed designs and task lists
- Task progress tracked via comments on phase issues (no per-task issues)
- Reference phase issue numbers in commit messages: `Ref #<phase-issue>`
- Use standardized label taxonomy (see `rdf github sync-labels`)

### Artifact Taxonomy

Working artifacts that are never committed:
- `CLAUDE.md` — project governance (generated by RDF)
- `PLAN*.md` — session-local phase plans
- `AUDIT.md` — audit pipeline output
- `MEMORY.md` — auto-memory persistence
- `.claude/` — tool-specific configuration
- `work-output/` — agent status files (ephemeral)

All excluded via `.git/info/exclude`, not `.gitignore`.

## Session Safety

- In `work-output/`, the `N` in all agent output filenames must be the integer phase number — descriptive labels break EM's file-existence verification
- EM must confirm expected output file paths exist (via `ls`) before reading — never assume subagent filenames without verification
